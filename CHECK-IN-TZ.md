# ТЗ: Автоматизированное заселение (воронка Check-in)

Проектное ТЗ на модуль **автоматизированного заселения** — от момента поступления
брони до момента открытия замка, для всех каналов бронирования. Документ рассчитан
на реализацию в Claude Code в этом монорепо, с **переиспользованием уже
построенного** (онлайн-регистрация `checkin/*`, цифровые ключи `keys/*`, доменный
гейт `canIssueKey`, уведомления `notifications/*`, публичные ссылки `links/*`,
задачи/уборка `ops/*`, RBAC, аудит).

Статус: **проектирование**. Ключевая идея — не строить три отдельных процесса, а
одна конфигурируемая **воронка заселения** (машина состояний на бронь), единая для
всех каналов; различаются только точка входа гостя и способ доставки ключа.

> Bitrix24 из ядра выведен (см. `CLAUDE.md`). Эскалации и «карточка гостя для
> сотрудника» — через собственные модули: задачи `ops/*` и коммуникации
> (`AI-COMMUNICATIONS-TZ.md`). Метод `crm.notifyKeyProblem` заменяется на создание
> `OpsTask`.

## 0. Контекст: что уже есть и что достраиваем

### 0.1 Уже построено (переиспользуем, НЕ переписываем)

| Блок | Где | Что умеет |
|---|---|---|
| Онлайн-регистрация | `apps/api/src/checkin/*`, модель `Checkin`, enum `CheckinStatus` (`NOT_STARTED→DRAFT→SUBMITTED→UNDER_REVIEW→APPROVED/REJECTED/NEEDS_FIX`) | Анкета гостя, паспорт (шифрование PII), загрузка скана в Object Storage, OCR, авто-проверка действительности, `submit`, действия сотрудника `approve/reject` |
| Цифровой ключ | `apps/api/src/keys/*`, модели `DigitalKey`/`Lock`, enum `KeyStatus`/`KeyType`/`LockTarget`, TTLock-порт | Выдача PIN + eKey, покрытие замков (личная/общие двери), удалённое открытие через шлюз, авто-назначение готового номера, авто-отзыв по окну (`keys.scheduler`, cron 15 мин) |
| Гейт выдачи ключа | `packages/domain/src/rules/key.ts` → `canIssueKey`, `computeKeyValidityWindow`, `shouldRevokeKey` | Чистая функция: бронь активна и не отменена, регистрация `APPROVED`, оплата/депозит выполнены, наступило окно (−30 мин / +30 мин, настраивается) |
| Ось шахматки | `pms/bookings/*`, enum `BookingStatus` (`PENDING→CONFIRMED→CHECKED_IN→CHECKED_OUT`, `NO_SHOW`, `CANCELLED`), `assertTransition` | Операционный статус номера; `checkIn()` (авто-назначение `roomId`), `markNoShow()`, `revertCheckIn()`, `reopenCheckout()` |
| Уведомления | `apps/api/src/notifications/*` (email/sms/push, `scenarios.ts`) | `notify(guestId, SCENARIO, payload)`; используется `CHECKIN_APPROVED`, `KEY_READY` |
| Публичные ссылки | `apps/api/src/links/public-link.service.ts` | Токен ≥128 бит, открытие без авторизации `/api/s/<token>`, отзыв, счётчик открытий (сейчас `kb_page`/`drive_file`) |
| Задачи | `apps/api/src/ops/*` (`OpsTask`) | Единый операционный таск с чек-листами/статусами/назначением — сюда идут эскалации |
| Оплата/депозит | `apps/api/src/payments/*`, `PaymentStatus`, `Deposit` | Статус оплаты и депозит брони |

### 0.2 Чего не хватает (это и есть предмет ТЗ)

1. **Конфигурируемая воронка** — сейчас шаги (регистрация → оплата → ключ) зашиты
   в коде. Нужен раздел, где владелец/менеджер **редактирует этапы и условия
   воронки, выбирает подключённые каналы коммуникации и описывает, как это
   работает** (см. §2). Это явное требование владельца.
2. **Оркестратор заселения** — сейчас ключ выдаётся только по запросу гостя
   (`keys.getForBooking/issue`), нет фонового двигателя, который ведёт бронь по
   воронке, шлёт напоминания по нужным каналам, авто-выдаёт ключ и авто-переводит
   `CONFIRMED → CHECKED_IN` (см. §6).
3. **Гость без аккаунта** — весь `checkin/*` и `keys/*` привязаны к `guestId`.
   Сценарии OTA и «прямая бронь без ЛК» требуют **гостевой ссылки заселения**
   (magic-link) поверх механизма `links/*`, чтобы человек без аккаунта прошёл
   регистрацию и получил ключ (см. §4).
4. **Стадия воронки как факт** — новая ось `CheckinFunnelStage` на бронь (см. §1, §9).

## 1. Две оси статусов и связь с шахматкой

Воронка заселения — это **не новые значения `BookingStatus`** и не замена
существующей машины состояний. Это **вторая, ортогональная ось**, живущая внутри
статуса `CONFIRMED` и кульминирующая в уже существующем переходе
`CONFIRMED → CHECKED_IN`.

```
Ось шахматки (BookingStatus, НЕ трогаем):
   PENDING ──▶ CONFIRMED ─────────────────────────▶ CHECKED_IN ──▶ CHECKED_OUT
                   │                                      ▲
Ось заезда (CheckinFunnelStage, новая; внутри CONFIRMED): │
                   └─ AWAITING → IDENTIFIED → REGISTERED → PAID → READY → KEY_ISSUED
                                                                            │
                        ключ выдан + наступило окно  ⇒  авто-вызов checkIn()  ┘
```

Принципы связи:

1. **Шахматка не меняется.** Администратор видит те же статусы/цвета. Овербукинг,
   `revert`, `reopen` работают как есть.
2. **Воронка — оркестратор над существующими полями, а не их дубликат.** Шлюзы
   читают то, что уже есть, и переиспользуют `canIssueKey`:
   - «регистрация» = `Checkin.status === APPROVED`;
   - «оплата/депозит» = `Booking.paymentStatus === PAID` (+ `Deposit`), с учётом
     `paymentRequired` объекта;
   - «номер назначен» = `Booking.roomId != null` (или `availability.autoAssignReadyRoom`);
   - «окно» = `computeKeyValidityWindow`.
3. **Самозаезд = программный вызов существующего `checkIn()`.** «Автоматическое
   заселение» — это автоматизация того же guard-перехода `CONFIRMED → CHECKED_IN`
   (с авто-назначением `roomId`), который сейчас делает администратор кнопкой. Второго
   механизма нет.
4. **NO_SHOW и revoke на своих местах.** Воронка не дошла до `READY` к дедлайну →
   существующий `markNoShow()`. `CHECKED_OUT`/`revertCheckIn` → `keys.revoke()`
   (авто-отзыв уже есть в `keys.scheduler`).

`CheckinFunnelStage` (новая ось, значения — производные, вычисляются оркестратором из
шлюзов; хранится как денормализованный кэш на брони для UI и отчётов):

| Стадия | Значит | Вычисляется из |
|---|---|---|
| `AWAITING` | бронь создана, гость ещё не в воронке | нет контакта/касания |
| `IDENTIFIED` | есть верифицированный контакт, гость привязан | телефон/email подтверждён |
| `REGISTERED` | онлайн-регистрация подтверждена | `Checkin.status = APPROVED` |
| `PAID` | оплата/депозит выполнены (если требуются) | `paymentStatus`, `Deposit` |
| `READY` | все шлюзы зелёные, окно наступило | `canIssueKey().allowed` |
| `KEY_ISSUED` | ключ(и) созданы в TTLock | есть `DigitalKey ACTIVE` |
| `NO_SHOW` / `CANCELLED` | терминальные | по оси шахматки |

## 2. Конструктор воронки заселения (главное новое требование)

Раздел админки, где **этапы и условия воронки редактируются данными, а не кодом**.
Владелец/менеджер: включает/выключает этапы, настраивает условия (шлюзы), выбирает
**подключённые каналы коммуникации** на каждом этапе, и пишет **описательный текст
«как это работает»** (виден и сотрудникам, и гостю в портале).

### 2.1 Модель конфигурации

- **`CheckinFunnel`** — воронка. Уровень: сеть (tenant) с переопределением на
  объект (`propertyId?`). Поля: `name`, `scope` (`TENANT`/`PROPERTY`), `active`,
  `description` (markdown «как устроено заселение у нас»), `default` (какая
  применяется, если у объекта своей нет).
- **`CheckinFunnelStage`** — этап воронки, упорядоченный (`order`). Поля:
  - `key` (машинный: `identification`/`registration`/`payment`/`key_issue`/… ; для
    типовых этапов) и `title` (человекочитаемый);
  - `enabled` — включён ли этап;
  - `required` — блокирующий ли шлюз (если `false` — этап информационный/опциональный,
    не мешает выдаче ключа);
  - `conditions` (Json) — набор условий-шлюзов из **словаря предопределённых
    условий** (не произвольный код): напр. `registration_approved`,
    `payment_paid`, `deposit_held`, `consents_signed`, `passport_valid`,
    `time_window_open`, `room_assigned`, `manual_admin_ok`. У условия могут быть
    параметры (напр. `payment_paid: { require: 'deposit_or_full' }`);
  - `channels` (String[]) — подключённые каналы коммуникации для этого этапа
    (см. §5): `push`, `sms`, `email`, `telegram`, `whatsapp`, `ota_messaging`,
    `guest_portal`. Выбираются из списка **фактически подключённых** каналов тенанта;
  - `notificationTemplateKey` — какой шаблон уведомления слать при входе в этап
    (реестр шаблонов, §5.2);
  - `reminderPolicy` (Json) — когда напоминать (напр. `[{ offsetHours: -24 },
    { offsetHours: -3 }]` относительно заезда) и по каким каналам;
  - `timing` (Json) — временные смещения этапа (напр. окно ключа −30/+30 мин
    переопределяет дефолт `computeKeyValidityWindow`);
  - `guestDescription` — текст «как это работает» для гостя (markdown, показывается
    в гостевом портале на этом шаге);
  - `staffNote` — заметка для сотрудника.
- **`CheckinFunnelChannel`** (или переиспользовать существующий реестр каналов
  тенанта из модуля коммуникаций) — какие каналы вообще подключены у тенанта и их
  креды/статус связи. Конструктор воронки предлагает выбирать только из подключённых.

### 2.2 Предустановленная воронка по умолчанию (seed)

Из коробки создаётся `default`-воронка, эквивалентная текущей зашитой логике (чтобы
ничего не сломать), которую дальше можно редактировать:

1. **Идентификация** (`identification`) — условие: верифицированный контакт;
   каналы: авто-выбор (push для ЛК, иначе sms/email/ota_messaging).
2. **Онлайн-регистрация** (`registration`, `required`) — условие `registration_approved`
   (+ `consents_signed`, `passport_valid`); каналы: напоминания.
3. **Оплата/депозит** (`payment`, `required`) — условие `payment_paid`/`deposit_held`
   (учёт `paymentRequired` объекта; для апартаментов/квартир может отличаться от отелей).
4. **Готовность и ключ** (`key_issue`, `required`) — условия `time_window_open` +
   `room_assigned`; действие: авто-выдача ключа + `KEY_READY`; переход
   `CONFIRMED → CHECKED_IN`.

### 2.3 Экран «Конструктор воронки заселения» (админка)

`apps/admin/src/app/pms/checkin-funnel/` (или в разделе «Настройки → Заселение»):

- список воронок (сетевая + переопределения по объектам), переключатель активной;
- **редактор этапов drag-and-drop** (порядок), тумблеры `enabled`/`required`;
- на каждом этапе: чек-боксы условий из словаря (с параметрами), мультиселект
  подключённых каналов, выбор шаблона уведомления + предпросмотр, редактор
  `guestDescription`/`staffNote`, настройка `reminderPolicy` и `timing`;
- поле `description` воронки — «как устроено заселение» (markdown);
- **предпросмотр**: как воронка выглядит для гостя (портал) по каждому из 3 сценариев;
- валидация: нельзя выключить обязательный по закону/безопасности шлюз (регистрация,
  оплата-если-required) без подтверждения; порядок этапов должен оставаться связным.
- права: `checkin_funnel_manage` (см. §12). Все изменения — в аудит.

## 3. Три сценария (точки входа в одну воронку)

Различие сценариев — только в шлюзах **идентификации/регистрации** и способе доставки
ключа. Дальше воронка общая (§2).

### 3.1 Сценарий 1 — гость с личным кабинетом (прямая бронь)

- Бронь рождается в нашем PMS/ЛК, `channel ∈ {WEBSITE, MOBILE_APP, MANAGER, PHONE,
  MESSENGER}`, `guestId` известен → этап «Идентификация» пройден сразу.
- Всё в приложении/веб-ЛК: регистрация с предзаполнением из профиля, оплата YooKassa,
  канал коммуникаций — **push**.
- Ключ — eKey/Bluetooth в приложении + PIN как fallback, удалённое открытие через шлюз.
- **Баллы D начисляются** (прямая бронь). Эталонный, самый быстрый путь.

### 3.2 Сценарий 2 — OTA (Booking/Островок/Avito и т.п.)

- Бронь приходит через **Channel Manager** (`channel = OTA`), `guestId` —
  синтетический/пустой контакт (OTA часто маскирует email/телефон).
- Стратегия дотянуться до гостя (по приоритету, задаётся в `channels` этапа
  идентификации):
  1. **OTA-messaging** (шаблон внутри OTA) со ссылкой на гостевой портал;
  2. если OTA отдаёт реальный контакт — **sms/email magic-link** (как в §3.3);
  3. fallback — код/инструкция на стойке (эскалация в `OpsTask`).
- Дальше — гостевой портал (§4), те же шлюзы регистрации/оплаты.
- **Баллы по умолчанию не начисляем** (OTA). Здесь предлагаем создать аккаунт:
  «в следующий раз бронируйте напрямую — вот баллы» → канал захвата гостя в свою базу.

### 3.3 Сценарий 3 — прямая бронь без личного кабинета (сайт/менеджер/звонок)

- Бронь в нашем PMS, аккаунта нет, но есть **реальный контакт** (телефон/email взят
  при бронировании).
- Отправляем **magic-link** (sms/email/мессенджер) → гостевой портал (PWA, без пароля).
- Гость проходит те же шлюзы. В конце — **мягкое** предложение «создать кабинет и
  получить баллы» (не форсируем; главный хук конверсии в ЛК).
- Ключ: PIN по sms или eKey через ту же ссылку.

Матрица (по умолчанию; всё переопределяется в конструкторе §2):

| | Идентификация | Канал | Доставка ключа | Баллы |
|---|---|---|---|---|
| 1. ЛК | сразу (`guestId`) | push | eKey в приложении + PIN | да |
| 2. OTA | достучаться | ota_messaging → sms/email | PIN по sms / портал | нет (по умолч.) |
| 3. Прямая без ЛК | magic-link | sms/email | PIN по sms / портал | да |

## 4. Гостевая ссылка заселения (magic-link, без аккаунта)

Расширяем механизм `links/public-link.service.ts` новым типом ресурса `checkin`,
привязанным к `bookingId`:

- **`GuestCheckinLink`** (или `PublicLink` с `resourceType = 'checkin'`): токен
  ≥128 бит, `bookingId`, `expiresAt` (по умолчанию — до конца окна выезда), `revokedAt`,
  `openCount`. Открытие без авторизации: `/api/s/checkin/<token>`.
- Токен даёт **ограниченную гостевую сессию** только на эту бронь: пройти
  онлайн-регистрацию (`checkin/*` — рефактор: сервис принимает либо `guestId`, либо
  контекст гостевой ссылки), внести оплату, получить/показать ключ (`keys/*` — тот же
  рефактор).
- Гостевой портал (PWA): страницы «Данные заезда» → «Онлайн-регистрация» → «Оплата»
  → «Ваш ключ», ведомые конфигом воронки (`guestDescription` на каждом шаге).
- Безопасность: токен одноразово-ротируемый при подозрении, rate-limit, журнал
  открытий; PII-скан паспорта — те же шифрование и `DocumentAccessLog`, что в
  `checkin/*`; после выезда ссылка авто-отзывается вместе с ключом.
- Рефактор существующего кода (аккуратно, с тестами): `CheckinService` и `KeysService`
  сейчас требуют `guestId`; вынести идентификацию гостя в единый «check-in principal»
  (аккаунт **или** гостевая ссылка), не ломая текущий guest-JWT-путь.

## 5. Каналы коммуникации

### 5.1 Подключённые каналы

Список каналов тенанта (реестр из модуля коммуникаций, `AI-COMMUNICATIONS-TZ.md`, +
существующий `notifications/*`): `push`, `sms`, `email`, `telegram`, `whatsapp`,
`ota_messaging`, `guest_portal`. Конструктор воронки (§2) позволяет выбрать на каждом
этапе только **фактически подключённые и проверенные** каналы (модалка «Настроить/
Проверить связь», как у платёжных систем в модуле «Финансы»).

### 5.2 Реестр шаблонов уведомлений

- Расширяем `notifications/scenarios.ts` сценариями воронки:
  `CHECKIN_INVITE` (приглашение в воронку/magic-link), `CHECKIN_REMINDER`,
  `PAYMENT_REMINDER`, `CHECKIN_APPROVED` (есть), `KEY_READY` (есть),
  `KEY_ISSUE_FAILED_ADMIN` (эскалация), `CHECKOUT_INFO`.
- Шаблоны редактируемы (текст на канал), с переменными (`{property}`, `{room}`,
  `{checkinTime}`, `{link}`, `{pin?}`). PIN в открытом канале — по политике объекта
  (можно запретить sms-PIN и слать только ссылку на портал).

### 5.3 Выбор канала оркестратором

При входе в этап оркестратор берёт `channels` этапа и шлёт по первому доступному для
данного гостя (есть push-токен? есть телефон? OTA-тред?), с фолбэком по списку.

## 6. Оркестратор заселения (двигатель воронки)

Новый сервис `apps/api/src/checkin/funnel/` + планировщик (паттерн `keys.scheduler`,
`@nestjs/schedule`, cron; Redis/BullMQ — опционально позже, по правилам DHP).

Обязанности:

1. **Инициация.** На событие «бронь `CONFIRMED`» (создание/подтверждение, в т.ч.
   импорт OTA) — вычислить стадию, при необходимости выпустить `GuestCheckinLink`,
   отправить `CHECKIN_INVITE` по каналам этапа идентификации.
2. **Продвижение по воронке.** По событиям (`Checkin.APPROVED`, `payment.paid`,
   `deposit.held`) и по расписанию — пересчитать `CheckinFunnelStage`, слать шаблон
   входа в этап и напоминания (`reminderPolicy`).
3. **Готовность и авто-выдача ключа.** Когда `canIssueKey().allowed` и наступило окно —
   авто-назначить готовый номер (`availability.autoAssignReadyRoom`), выдать ключ
   (`keys.issue`), послать `KEY_READY`, и **авто-перевести `CONFIRMED → CHECKED_IN`**
   (тот же guard-переход `pms-booking.checkIn()`), если объект настроен на автозаезд.
   Иначе — оставить на подтверждение стойки.
4. **Дедлайны.** Если к дедлайну (напр. заезд + N часов) воронка не дошла до `READY` —
   `markNoShow()` по политике объекта + эскалация.
5. **Эскалации — в `OpsTask`, не в Bitrix24.** Проблема выдачи ключа
   (замена `crm.notifyKeyProblem`), не пройдена регистрация/не оплачено к заезду,
   TTLock недоступен → создаётся задача администратору/стойке с контекстом брони.
6. **Выезд.** На `CHECKED_OUT`/`revertCheckIn` — отзыв ключей (уже есть) + отзыв
   `GuestCheckinLink` + `CHECKOUT_INFO`.

Идемпотентность: все действия оркестратора идемпотентны (повторный тик не плодит
ключи/уведомления) — как уже сделано в `keys.issue` (проверка `ACTIVE`-ключа).

---

## 7. Жизненный цикл цифрового ключа (в основном готов)

Переиспользуем `keys/*` полностью. Дополнения:

- Окно действия и типы доступа (PIN / Bluetooth eKey / через gateway / резервный код
  администратора / доступ персонала) — как в `CLAUDE.md`. `timing` этапа `key_issue`
  переопределяет дефолтное окно.
- **После проживания доступ удаляется** — уже реализовано (`keys.autoRevokeExpired`,
  cron 15 мин; `revoke` удаляет пароль и обнуляет `pinEncrypted`).
- Весь жизненный цикл логируется (`DigitalKeyLog`).
- Привилегии раннего заезда/апгрейда — **только отели, не апартаменты** (тип объекта),
  учитывается в условиях этапа и окне ключа.

## 8. Fallback-и и крайние случаи (иначе гость встанет у двери)

- Регистрация/оплата не пройдены к заезду → ключ не выдаётся (`canIssueKey`
  вернёт reasons), напоминания гостю + `OpsTask` на стойку.
- **TTLock/gateway офлайн** → ключ `FAILED` (уже обрабатывается), резервный PIN
  администратора, `OpsTask`. Гость видит понятное сообщение, не «ошибка».
- **Депозит** — баллами платить нельзя (только деньги); без него шлюз `payment` красный.
- Готовый свободный номер не найден к моменту готовности → сообщение «ожидайте
  назначения» (уже есть в `keys.getForBooking`) + задача на стойку.
- OTA не отдал контакт и нет OTA-messaging → полностью ручной заезд через стойку
  (воронка помечает `manual`), ключ выдаёт администратор (`keys.adminIssue`).
- Гость по magic-link потерял ссылку → повторная отправка/ротация токена сотрудником.

## 10. API

**Гостевой портал (без авторизации, по токену):**
- `GET /api/s/checkin/:token` — контекст брони + текущий шаг воронки + `guestDescription`.
- `POST /api/s/checkin/:token/registration` — сохранить/submit анкету (проксирует в `checkin/*`).
- `POST /api/s/checkin/:token/passport` — загрузить скан (шифрование, как есть).
- `POST /api/s/checkin/:token/pay` — инициировать оплату (YooKassa).
- `GET /api/s/checkin/:token/key` — состояние ключа/PIN (проксирует в `keys/*`).
- `POST /api/s/checkin/:token/key/issue` / `.../open` — выдать/удалённо открыть.

**ЛК (авторизованный гость):** существующие `checkin/*` и `keys/*` — без изменений
(добавляется единый «check-in principal», §4).

**Админка:**
- CRUD `/api/admin/checkin-funnels` (+ этапы) — конструктор воронки (`checkin_funnel_manage`).
- `GET /api/admin/checkin/queue` — очередь заездов/стадий (для стойки).
- `POST /api/admin/checkin/:bookingId/link` — выпустить/переслать magic-link.
- `POST /api/admin/checkin/:bookingId/key/issue|revoke` — ручная выдача/отзыв (есть `adminIssue`).

## 13. Что переиспользуем и НЕ строим заново

- Онлайн-регистрацию (`checkin/*`), ключи (`keys/*`), гейт `canIssueKey`, окно ключа,
  авто-отзыв, авто-назначение номера, TTLock-порт — **как есть**, достраиваем сверху.
- Уведомления (`notifications/*`) — добавляем сценарии/шаблоны, не переписываем канал.
- Публичные ссылки (`links/*`) — расширяем типом `checkin`.
- Оплату/депозит (`payments/*`), задачи (`ops/*`, вместо Bitrix), аудит, RBAC.
- Планировщик — паттерн `keys.scheduler` (cron `@nestjs/schedule`).

## 14. Безопасность и ПДн

- Паспорт/скан — только шифрование PII (`CryptoService.encryptPii`/`encryptBuffer`),
  Object Storage, `retentionUntil`, `DocumentAccessLog` — уже соблюдено, распространить
  на гостевой magic-link-путь.
- Magic-link токен ≥128 бит, срок жизни = окно брони, отзыв при выезде/подозрении,
  rate-limit, журнал открытий.
- PIN не показываем вне окна действия (уже так в `keys.buildView`); политика
  «PIN в sms» — настраиваемая (может быть запрещена, только портал).
- Все действия воронки/ключа/сотрудника — в аудит.

## 15. Порядок реализации (спринты)

1. **Ось воронки + связь с шахматкой** — `funnelStage` на брони, вычисление из шлюзов,
   панель «Заселение» в карточке брони (только чтение). Без автоматизации.
2. **Конструктор воронки** — модели `CheckinFunnel/Stage`, seed default-воронки,
   админ-раздел (§2.3), словарь условий и каналов.
3. **Оркестратор** — планировщик + событийное продвижение, напоминания по каналам,
   авто-выдача ключа, авто-`CHECKED_IN`, дедлайны/NO_SHOW, эскалации в `OpsTask`
   (замена `crm.notifyKeyProblem`).
4. **Гостевой magic-link портал** — расширение `links/*`, рефактор `checkin/keys` на
   единый principal, PWA-страницы, сценарии OTA (§3.2) и прямой-без-ЛК (§3.3).
5. **Каналы и шаблоны** — реестр шаблонов, OTA-messaging, предпросмотр в конструкторе.
6. **Полировка** — очередь заезда для стойки, отчёты по воронке, edge-cases (§8).
