# ТЗ: Аудио/видео-звонки сотрудников (§2.2, v2)

Проектное ТЗ на блок звонков внутреннего мессенджера. **Только план — кода нет.**
Задача документа: выбрать технологию и архитектуру, чтобы владелец принял решение до
разработки. Опирается на `AI-COMMUNICATIONS-TZ.md` §2.2 и на уже готовый мессенджер
(`apps/api/src/staff-chat/*`, SSE-realtime, admin-JWT, `infra/terraform` под Yandex Cloud).

Статус: **проектирование, решения владельца приняты 2026-07-11** (см. §0). Мессенджер
(§2) функционально закрыт; звонки — единственный оставшийся пункт, отнесён ТЗ к v2,
т.к. требует отдельной инфраструктуры.

## 0. Решения владельца (2026-07-11) — ЗАФИКСИРОВАНО

1. **Группы — сразу.** MVP включает групповые звонки, не только 1:1 → нужен полноценный
   SFU с самого старта (**LiveKit**), не «P2P без сервера».
2. **Запись — не нужна.** Без Egress; серверная запись не закладывается.
3. **Мобайл — сразу.** Звонки в Expo-приложении с самого начала. **Следствие:**
   `apps/mobile` переводим с Expo Go на **dev-client/prebuild** — LiveKit React Native SDK
   в Expo Go не работает (нужна нативная сборка). Разовая, но заметная работа.
4. **Постоянная ВМ под SFU — принято.** Ежемесячная аренда ВМ + трафик; стартуем с
   небольшой машины, масштабируем под нагрузку.
5. **Доступ — все могут звонить.** Отдельного права НЕ вводим — используем существующее
   **`staff_chat`** (кто в мессенджере, тот и звонит).
6. **Шифрование — обычное (DTLS-SRTP).** E2EE не делаем (медиа-сервер в нашем контуре РФ).

## 2. Почему это отдельный инфраструктурный блок

Звонки — не «ещё один REST-эндпоинт». WebRTC требует трёх вещей, которых у нас нет:

1. **Сигналинг** — обмен SDP-офферами/ответами и ICE-кандидатами, чтобы два браузера
   договорились о медиасессии. (У нас есть готовый канал — SSE + REST, см. §6.)
2. **STUN/TURN** — обход NAT. STUN сообщает браузеру его внешний адрес; **TURN**
   ретранслирует медиа, когда прямое соединение невозможно (симметричный NAT,
   корпоративные файрволы). В офисных сетях TURN практически обязателен.
3. **Медиа-сервер (SFU)** — для групп: вместо N² P2P-соединений браузер шлёт один
   поток на сервер, тот раздаёт остальным. Для 1:1 можно без сервера (P2P), для групп —
   нельзя.

Медиа **никогда не идёт через наш NestJS-backend** — только P2P или через SFU. Backend
занимается лишь контролем звонка (кто кому звонит, токены доступа, журнал).

| Клиентские SDK | Отличные: web (`livekit-client`), React, **React Native** (для Expo-мобайла), iOS/Android | lib-jitsi-meet / iframe | Только браузерный WebRTC + твой протокол | Голый RTCPeerConnection |

| Сложность внедрения | **Низкая-средняя** | Средняя-высокая | **Высокая** | Средняя (но тупик на группах) |
| Лицензия | Apache-2.0 | Apache-2.0 | ISC | — |
| Ресурсы (SFU) | ~1 vCPU на ~20–40 видеоучастников | Похоже | Похоже | н/д |

**Вывод по вариантам:**
- **mediasoup** — максимум контроля, но максимум работы (пишем оркестрацию комнат сами).
  Оправдан при нестандартных требованиях; у нас их нет.
- **Jitsi** — отличен для больших публичных конференций, но тяжёлый стек из 3–4 сервисов
  и более «коробочный» (свой UI). Для встраивания в наш мессенджер избыточен.
- **Raw WebRTC + coturn** — годится ТОЛЬКО для 1:1. Тупик, как только нужны группы.
- **LiveKit** — золотая середина: один сервер, JWT-токены (совпадает с нашей моделью
  авторизации), готовые SDK под web + React Native (важно для Expo-мобайла из
  `apps/mobile`), встроенный TURN, Egress для записи. Масштабируется от 1:1 до групп
  без смены технологии.

## 4. Рекомендация

**LiveKit, self-hosted на Yandex Cloud.** Обоснование:

1. **Одна технология на весь путь** — 1:1 сегодня, группы/экран/запись потом, без
   переписывания.
2. **Ложится на наш стек**: токены доступа = JWT (у нас уже admin-JWT + `livekit-server-sdk`
   на бэкенде выпускает токен с identity=adminId, room, grants). Сигналинг вызова —
   через **готовый SSE** (`StaffChatEvents`), рингтон — через **готовые** push+звук
   мессенджера. Мобайл — `@livekit/react-native` (Expo dev-client).
3. **Self-hosted на Yandex Cloud** — соответствует решению владельца (данные/медиа в РФ,
   есть `infra/terraform`). Никаких внешних SaaS, ПДн не покидают периметр.
4. **Медиа не через наш backend** — SFU отдельный; наш API не становится узким местом.

## 5. Архитектура (интеграция с нашим стеком)

```
Браузер/приложение сотрудника
  │  1) POST /staff-calls/start {chatId}         (наш NestJS API)
  │  2) ← { callId, room, token(LiveKit), url }
  │  3) SSE 'call:incoming' → другому участнику  (наш StaffChatEvents)
  │  4) участник: POST /staff-calls/:id/accept → свой token
  │
  ▼ 5) WebRTC (livekit-client) ↔ LiveKit SFU (ws + медиа, DTLS-SRTP)
LiveKit SFU (ВМ Yandex Cloud, публичный IP)
  ├─ встроенный TURN (TLS 5349) / внешний coturn
  └─ (опц.) Egress → Yandex Object Storage (запись)

Наш backend НЕ проксирует медиа — только: токены, жизненный цикл звонка, журнал, SSE.
```

- **livekit-server** — ВМ на Yandex Cloud (2–4 vCPU / 4–8 ГБ на старте), публичный IP,
  открытые порты (WS 7880, WebRTC UDP-диапазон, TURN 3478/5349). Разворачивается через
  `infra/terraform` (добавить ресурс ВМ + группу безопасности) + docker/compose.
- **coturn** (или встроенный TURN LiveKit) — на том же/соседнем публичном IP. Для
  корпоративных NAT — обязателен; проще стартовать со **встроенного TURN LiveKit (TLS)**.
- **Наш модуль** `apps/api/src/staff-calls/` — контроль звонков (см. §6).
- **Фронт** `apps/admin` (и позже `apps/mobile`) — `livekit-client`/`@livekit/react-native`:
  `getUserMedia`, публикация/подписка треков, оверлей звонка (модалка: превью, кнопки
  mute/камера/экран/завершить, сетка участников для групп).

## 6. Бэкенд: сигналинг вызова через SSE (без нового WebSocket)

ТЗ §2.2 упоминал «сигналинг через WebSocket», но у нас **уже есть** per-user realtime —
SSE (`StaffChatEvents`). Медиа-сигналинг (SDP/ICE) берёт на себя LiveKit (его WS
клиент↔SFU). Нам нужен только **сигналинг вызова** (кто звонит / принято / отклонено /
завершено) — он прекрасно ложится на наш SSE. Отдельный WebSocket не нужен.

**Новые kind SSE-событий** (расширяем `StaffChatEvent`): `call:incoming`, `call:accepted`,
`call:declined`, `call:ended`, `call:participant` (join/leave).

**Эндпоинты** `apps/api/src/staff-calls/` (под `AdminAuthGuard` + право `staff_chat`):
- `POST /staff-calls/start` `{ chatId }` → создаёт `Call` (status RINGING) + room LiveKit
  (`room = call_<uuid>` для 1:1, `= chat_<chatId>` для группы), выпускает токен инициатору,
  рассылает `call:incoming` остальным участникам чата. Ответ: `{ callId, room, url, token }`.
- `POST /staff-calls/:id/accept` → токен участнику, status ACTIVE, SSE `call:accepted`.
- `POST /staff-calls/:id/decline` → SSE `call:declined` (для 1:1 → сразу end).
- `POST /staff-calls/:id/leave` → участник вышел; если пусто — status ENDED, SSE `call:ended`.
- (опц.) вебхук LiveKit → наш API (participant_joined/left, room_finished) для точного
  журнала.

**Выпуск токена**: `livekit-server-sdk` (`AccessToken`) с `identity = adminId`,
`name = имя`, grant `roomJoin`, `room`, `canPublish/canSubscribe`. Ключ/секрет LiveKit —
в env (`LIVEKIT_API_KEY/SECRET/URL`), как `DEEPSEEK_*`/`TELEGRAM_*`.

## 7. Модель данных (Prisma)

```
enum StaffCallKind   { AUDIO VIDEO }
enum StaffCallStatus { RINGING ACTIVE ENDED MISSED DECLINED }

model StaffCall {
  id           String @id @default(uuid())
  tenantId     String
  chatId       String?          // связь с чатом (группа/DM)
  kind         StaffCallKind
  room         String @unique   // имя комнаты LiveKit
  status       StaffCallStatus @default(RINGING)
  startedById  String
  startedAt    DateTime @default(now())
  endedAt      DateTime?
  participants StaffCallParticipant[]
  @@index([tenantId, startedAt])
}

model StaffCallParticipant {
  id       String @id @default(uuid())
  callId   String
  userId   String
  joinedAt DateTime?
  leftAt   DateTime?
  call     StaffCall @relation(fields: [callId], references: [id], onDelete: Cascade)
  @@unique([callId, userId])
}
```

Даёт журнал/историю звонков (в чате — плашка «Звонок 3:12» / «Пропущенный звонок»,
как в мессенджерах) и метрики.

## 8. Безопасность и ПДн

- **Шифрование транзита**: LiveKit — DTLS-SRTP по умолчанию. SFU видит медиа
  (не E2EE) — для внутренних звонков сотрудников приемлемо. **E2EE** (insertable
  streams) — опционально позже, ценой отказа от серверной записи/обработки.
- **Токены доступа** — короткоживущие (напр. 10–15 мин, на вход в комнату), привязаны к
  identity=adminId и конкретной комнате; грант проверяет членство в чате.
- **Изоляция**: имя комнаты включает tenant/chat; токен выдаётся только участнику чата.
- **Запись** (если включат) — только с явным уведомлением всех участников (баннер
  «Идёт запись») и согласием; хранение в Yandex Object Storage с ограничением доступа и
  сроком (152-ФЗ). Это отдельное юр-решение владельца.
- Медиа и записи **не покидают периметр** (self-hosted РФ).

## 9. Хостинг и стоимость (грубая оценка, Yandex Cloud)

- **1 ВМ под LiveKit** (старт: 2 vCPU / 4 ГБ, публичный IP) — растёт по числу
  одновременных участников (~1 vCPU на 20–40 видеопотоков). Встроенный TURN — там же.
- **Трафик**: видео ~1–2 Мбит/с на поток; аудио ~40 кбит/с. Основная статья — исходящий
  трафик SFU при группах.
- **Egress-запись** (если понадобится) — отдельная ВМ/контейнер + Object Storage.
- Порядок: одна недорогая ВМ + публичный IP + трафик; на десятки одновременных
  сотрудников — умеренно. Точную смету считать под нагрузку.

| Фаза | Содержание | Инфра | Backend | Фронт |
|---|---|---|---|---|
| 1 | **1:1 аудио**: LiveKit+встроенный TURN, токены, SSE-вызов, оверлей звонка, журнал | LiveKit ВМ + terraform | модуль `staff-calls` (start/accept/decline/leave, токены, `StaffCall*`) | livekit-client, UI входящего/исходящего, mic on/off |

несколько дней; **фронт-UI звонка** — несколько дней. Фаза 1 — самостоятельный, ценный
результат.
