# Passport OCR — self-hosted сайдкар распознавания паспорта РФ

Небольшой сервис на FastAPI + PaddleOCR, который распознаёт скан паспорта
**локально** (152-ФЗ: изображение никуда не уходит) и отдаёт поля для
автозаполнения онлайн-регистрации. Гость затем **подтверждает/правит** поля —
это часть регламента, OCR не заменяет проверку.

## Контракт (согласован с backend `HttpPassportAdapter`)

```
GET  /health      -> { "status": "ok" }

POST /recognize
  body: { "image": "<base64 скана>", "contentType": "image/jpeg" }
  resp: {
    "fields": {
      "series": "4005", "number": "123456",
      "lastName": "Иванов", "firstName": "Иван", "middleName": "Иванович",
      "birthDate": "1990-05-01",
      "issuedBy": "...", "issuedDate": "2010-05-20"
    },
    "confidence": 0.9,          // 0..1
    "source": "mrz" | "page",
    "note": "Распознано по MRZ."
  }
```

Любое поле может быть пустым — backend отдаёт результат гостю на подтверждение.

## Автозапуск через деплой (рекомендуется)

Деплой-пайплайн сам поднимает и обновляет сайдкар — см. `infra/deploy/passport-ocr.sh`
(вызывается из `.github/workflows/deploy.yml` после основного деплоя). Условия:

1. На сервере установлен **docker**.
2. В `apps/api/.env` (на сервере) стоит `PASSPORT_PROVIDER=http` — это **единственный
   переключатель**: он и активирует адаптер, и включает автоподъём контейнера.

Дальше при каждом деплое: образ пересобирается **только если** менялись исходники
сайдкара (по хэшу), контейнер перезапускается на `127.0.0.1:8077`. Шаг **никогда не
роняет деплой** (нет docker / ошибка сборки → тихий пропуск, основной деплой идёт).
По умолчанию (`PASSPORT_PROVIDER=mock`) сайдкар не поднимается.

## Запуск вручную (Docker)

```bash
cd services/passport-ocr
docker build -t dha-passport-ocr .
# слушаем только localhost сервера — наружу НЕ публикуем
docker run -d --restart=unless-stopped --name dha-passport-ocr \
  -p 127.0.0.1:8077:8077 dha-passport-ocr
curl -s http://127.0.0.1:8077/health   # {"status":"ok"}
```

Первая сборка тянет модели PaddleOCR (~несколько сотен МБ) и «запекает» их в
образ, поэтому первый `docker build` идёт несколько минут; рантайм — быстрый.

### Без Docker (venv)

```bash
cd services/passport-ocr
python3.11 -m venv .venv && . .venv/bin/activate
pip install -r requirements.txt
uvicorn app:app --host 127.0.0.1 --port 8077 --workers 1
```

Для автозапуска — systemd-юнит или pm2 (`pm2 start "uvicorn app:app --host 127.0.0.1 --port 8077" --name dha-passport-ocr`).

## Включение на стороне backend

Сервис распознавания уже реализован в API — нужно лишь **переключить провайдера**
и указать адрес. В `.env` API (на сервере) выставить:

```
PASSPORT_PROVIDER=http
PASSPORT_OCR_URL=http://127.0.0.1:8077
# проверка действительности по списку недействительных МВД (Dadata «Проверка паспорта»)
DADATA_API_KEY=<ключ>
DADATA_SECRET=<секрет>
```

После этого перезапустить `dha-api` (pm2 restart dha-api). Без `DADATA_*` распознавание
работает, а проверка действительности отдаёт вердикт `MANUAL` (ручная проверка).

## Ресурсы и приватность

- CPU-инференс (GPU не требуется). Ориентир: ~1–2 ГБ RAM на инстанс, распознавание
  1–3 c на скан. Держим 1 воркер; при потоке — несколько реплик за балансировкой.
- Сервис **не логирует изображения** и не пишет их на диск; работает in-memory.
- Публиковать наружу нельзя — только внутренний адрес `127.0.0.1:8077` для API.

## Точность

Надёжнее всего извлекаются ФИО и дата рождения из **MRZ** (2 строки внизу 2-й
страницы). Серия/номер берутся по видимому тексту и могут распознаться неточно —
поэтому обязателен шаг подтверждения гостем. Качество зависит от фото (свет,
резкость, отсутствие бликов). Тестировать на реальных сканах после деплоя.
