# ТЗ: Система коммуникаций D H&A + два AI-агента

Техническое задание для Claude Code на разработку единой системы коммуникаций
между сотрудниками, отделами и гостями сети **D Hotels & Apartments**, включая
двух AI-агентов: копилот сотрудника и агент для гостей.

Составлено на основе `Промт.docx`, `CLAUDE.md` и обсуждения с владельцем
2026-07-08. Язык проекта — русский.

## 1. Архитектурные принципы (не нарушать)

```
Гость / сотрудник → канал (web / app / Telegram / админка)
                          ↓
                 backend D H&A (единственный владелец ключей и логики)
                    ├─ LLM-провайдер (порт → Claude | YandexGPT)
                    ├─ RAG / база знаний
                    ├─ инструменты (tools) под RBAC и деперсонализацией
                    ├─ PMS / лояльность / платежи / ключи / уведомления
                    └─ внутренние задачи и лента эскалаций (в нашем ПО, без внешней CRM)
```

1. **Клиенты никогда не обращаются к LLM, YooKassa, TTLock напрямую.**
   Всё — через backend. Причины те же, что в CLAUDE.md: ключи, ПДн, бизнес-логика,
   единая база гостей, независимость от вендора.
2. **LLM-провайдер — через порт/адаптер** (по образцу `Bitrix24Port`,
   `BnovoPort`). Никакой прямой зависимости бизнес-логики агента от конкретного
   SDK. Переключение Claude ↔ YandexGPT ↔ др. — конфигом (`LLM_PROVIDER`).
3. **	 через
   существующий `pms/bookings` с проверкой доступности в PG-транзакции и
   `Idempotency-Key`. Агент не пишет в брони в обход PMS-логики.
4. **ПДн не уходят в модель — на двух уровнях.** (а) *Данные бэкенда:* инструменты
   возвращают модели обезличенные представления (ID, суммы, даты, категории);
   паспорт/ФИО/телефон/скан остаются в backend. (б) *Входящий текст гостя:* гость
   может сам написать ФИО/телефон — перед отправкой в модель применяется
   маскирование ПДн (§8); идентификация/паспорт собираются через защищённую форму,
   не через модель.
5. **Всё логируется.** Каждый вызов инструмента, каждое сообщение, токены и
   стоимость — в журнал (аудит + ПДн + метрики).
6. **Действия с последствиями — с подтверждением** (сотрудника или гостя).
   Модель предлагает, человек/гость подтверждает.

## 2. Слой A — внутренний мессенджер сотрудников

**Работаем в нашем собственном ПО, локально — без внешней CRM (Bitrix24 из ТЗ
исключён 2026-07-08).** Строим полноценный корпоративный мессенджер в админке для
общения сотрудников и отделов + задачи + эскалации.

### 2.1. Чат-функционал (как в популярных мессенджерах)
- **Личные (1:1) и групповые чаты** — группы по отделам/объектам/темам; роли в
  группе (владелец/админ/участник); публичные и приватные группы; аватар/название.
- **Сообщения:** текст, эмодзи, **реакции** на сообщения, ответы (reply/quote),
  пересылка (forward), редактирование и удаление, упоминания `@сотрудник`,
  треды/ветки, черновики.
- **Медиа и файлы:** загрузка изображений/документов/видео, превью, галерея чата,
  раздел «медиа и файлы» по чату (S3-совместимое хранилище).
- **Голосовые сообщения** (запись/воспроизведение); видео-кружки — опц.
- **Аудио/видео-звонки** между сотрудниками — 1:1 и групповые (§2.2).
- **Поиск** — полнотекстовый по сообщениям и чатам; фильтры (автор, дата, тип
  вложения).
- **Разделы/папки** — группировка чатов («Избранное», «Непрочитанное», по отделам,
  пользовательские папки).
- **Закреплённые сообщения** (pin) в чате; **избранные/сохранённые сообщения**
  (личная коллекция сотрудника).
- **Настройка уведомлений** — per-chat mute, «только упоминания», расписание/DND,
  каналы (push/email), звук.
- **Статусы:** доставлено/прочитано (read receipts), индикатор набора (typing),
  присутствие (онлайн/офлайн/был в сети), счётчики и отметка «непрочитано»,
  напоминания по сообщению.
- **Стикеры/GIF** (опц.), предпросмотр ссылок.

### 2.2. Аудио/видео-звонки
- 1:1 и групповые; демонстрация экрана (опц.), запись (опц., с уведомлением
  участников).
- Технология: **WebRTC** + self-hosted SFU/TURN (напр. LiveKit или Jitsi на Yandex
  Cloud); сигналинг — через наш backend (WebSocket), медиа P2P/через SFU. Отдельная
  инфраструктура — этап v2 (§13).

### 2.3. Задачи и уведомления
- **Задачи между отделами** — `pms/operations` (инженер/уборка),
  `warehouse/requests` (склад) + лёгкий модуль **внутренних задач/поручений**
  (`internal-tasks`): назначение на сотрудника/отдел, статусы, комментарии, связь с
  чатом/сообщением.
- **Уведомления** — через `notifications` (push/email/SMS + `scenarios.ts`): новое
  сообщение/упоминание, назначена задача, эскалация от гостевого агента, требуется
  подтверждение действия агента, входящий звонок.

Эскалации гостевых диалогов приходят в тот же интерфейс (лента эскалаций §4.7);
оператор может делегировать диалог другому сотруднику (§4.8).

## 3. Слой B — AI-копилот сотрудника

### 3.1. Назначение
Чат-помощник в **админ-панели**: отвечает на вопросы по работе SaaS, по базе
знаний (регламенты), по живым данным (доступность, брони, гости, склад,
аналитика) и **выполняет поручения** — создаёт задачи/заявки/уведомления и
вносит изменения строго в пределах прав роли сотрудника.

### 3.2. Канал
Встроенный чат в `apps/admin` (Next.js). Доступ — по новому праву `ai_copilot`.

### 3.3. Модель
Через `LlmPort`. Дефолт (решение 2026-07-08) — **DeepSeek (китайское облако, hosted
API)**: V4 Flash для роутинга/простых ответов, V4 Pro для основной работы, R1 — для
сложного многошагового рассуждения. Системный промпт и ядро базы знаний держим
стабильным префиксом (у DeepSeek дешёвое чтение кэша). Обоснование — §14.1;
ПДн-последствия облака — §8.

### 3.4. База знаний (RAG)
- Раздел «База знаний» в админке (CRUD, право `ai_kb`): документы с полем
  `scope` = `staff | guest | both`.
- Индексация: документ → чанки → эмбеддинги (через `EmbeddingPort`, см. §5.2) →
  векторный поиск (pgvector). Под каждый вопрос подтягиваются top-k чанков
  области `staff`/`both`.
- База знаний **не содержит ПДн** (это регламенты/инструкции), поэтому её
  эмбеддинги можно считать любым провайдером.

### 3.5. Инструменты (tools)
Каждый инструмент помечен требуемым правом. **Агент физически может вызвать
только те инструменты, на которые у сотрудника есть право** (`AdminUser` roles →
`permissions.ts`). Реестр инструментов фильтруется по правам перед каждым
запросом к модели.

**Read (чтение):**
| Инструмент | Требуемое право | Что делает |
|---|---|---|
| `kb_search(query)` | `ai_copilot` | поиск по базе знаний (staff/both) |
| `search_availability(filters)` | `pms_availability` | свободные номера/даты |
| `find_booking(query)` | `pms_bookings` | поиск брони |
| `get_booking(id)` | `pms_bookings` | карточка брони (обезличенная) |
| `get_guest_summary(id)` | `guests` | сводка по гостю без чувствительных ПДн (без скана/номера паспорта) |
| `get_loyalty(guestId)` | `guests` | уровень/баллы |
| `warehouse_balance(filters)` | `wh_balances` | остатки |
| `analytics_query(metric, period)` | `analytics` | показатели |

**Write (действия — с подтверждением сотрудника):**
| Инструмент | Требуемое право | Что делает |
|---|---|---|
| `create_maintenance_task(...)` | `pms_maintenance` | заявка инженеру |
| `create_housekeeping_task(...)` | `pms_housekeeping` | задача уборки |
| `create_warehouse_request(...)` | `wh_requests` | складская заявка |
| `add_booking_note(id, text)` | `pms_bookings` | комментарий к брони |
| `send_notification(target, scenario)` | `ai_copilot` | уведомление гостю/сотруднику |
| `create_internal_task(...)` | `ai_copilot` | внутренняя задача сотруднику/отделу (модуль `internal-tasks`) |

**Draft-only (черновики, ничего не меняют):**
`draft_guest_reply(context)`, `draft_document(kind, context)` — возвращают текст,
сотрудник копирует/правит сам.

### 3.6. Guardrails
- Любой write-инструмент → модель формирует предложение → UI показывает
  «Подтвердить / Отменить» → выполнение только после подтверждения сотрудника.
- Вне прав роли инструмент недоступен (тест: копилот front-desk не может создать
  складскую заявку).
- Все вызовы — в `AiToolAudit` (кто, что, аргументы обезличенно, результат).

## 4. Слой C — AI-агент для гостей

### 4.1. Назначение
«AI-администратор»: отвечает гостям, использует базу знаний и живые данные SaaS
(тарифы, категории, условия бронирования, доступность), **готовит** бронь и
оплату (гость подтверждает оплатой), присылает документы, помогает с лояльностью
и ключом, при сложном/чувствительном вопросе — **подключает администратора**.

### 4.2. Каналы (MVP)
- **Виджет на сайте** и в веб-ЛК (`apps/web`).
- **Встроенный чат в мобильном приложении** (`apps/mobile`, Expo).
- **Telegram-бот** (adapter, §9).

Все каналы приводятся к единому `AiConversation` через `ChannelPort` (нормализация
входящих/исходящих). v2: WhatsApp, Avito.

### 4.3. Модель
Через `LlmPort`. Дефолт (решение 2026-07-08) — **DeepSeek (китайское облако, hosted
API)**, V4 Flash/Pro. **Важно:** endpoint в материковом КНР → **маскирование
входящего текста (§8) обязательно**, плюс согласие гостя на трансграничную передачу
и уведомление РКН. Резерв за тем же портом — YandexGPT 5 Pro (managed, РФ, без
трансграничности) или self-host (если позже вернёмся к «данные только в РФ»).

### 4.4. Логика диалога (pipeline)
```
Входящее сообщение (любой канал)
  1. нормализация → AiConversation (+ идентификация гостя, если авторизован)
  2. определить язык + намерение (Haiku)
  3. собрать контекст:
       • релевантные чанки базы знаний (scope guest/both)
       • живые данные SaaS через инструменты (тарифы, доступность, условия)
       • история диалога (из нашей БД) + prompt cache стабильного префикса
  4. модель рассуждает:
       • простой вопрос            → ответ из контекста
       • нужно действие            → вызов инструмента (готовит; деньги/ПДн — с гардами)
       • сложно/чувствительно/спор → escalate_to_admin
  5. ответ гостю (стриминг)
  6. запись сообщения + токенов + стоимости; при эскалации — в ленту §2
```

### 4.5. Инструменты (готовит — гость подтверждает; деперсонализация)
| Инструмент | Что делает | Гарды |
|---|---|---|
| `kb_search(query)` | база знаний (guest/both) | — |
| `find_properties(filters)` | подбор объектов | — |
| `check_availability(roomTypeId, dates, guests)` | доступность | через `pms/availability` |
| `quote_price(roomTypeId, dates, guests, ratePlanId?, promocode?)` | расчёт цены и условий | цена только из `pms/rates`, не выдумывается |
| `list_extras(propertyId)` | доп. услуги | апселл |
| `create_booking_hold(payload, idempotencyKey)` | бронь-hold (не подтверждена) | через `pms/bookings`, PG-транзакция, овербукинг невозможен |
| `issue_payment_link(bookingId)` | ссылка YooKassa | оплата = подтверждение брони |
| `get_booking_status(ref)` | статус брони гостя | только для авторизованного гостя |
| `send_confirmation_docs(bookingId)` | ваучер/подтверждение | через `notifications` (email/PDF) |
| `get_loyalty_balance()` | баллы/уровень текущего гостя | право = сам гость |
| `request_early_checkin(bookingId)` / `request_late_checkout(bookingId)` | ранний заезд/поздний выезд | **только тип объекта = отель**, не апартаменты |
| `get_key_help(bookingId)` | статус/инструкция по ключу TTLock | выдача ключа — только по правилам `keys` (регистрация, оплата, окно) |
| `escalate_to_admin(reason, summary)` | передать человеку | ставит диалог в ленту эскалаций админки (operator inbox) + уведомление |

**Не инструменты (только эскалация):** отмена брони, возврат средств, изменение
уже оплаченной брони, любые споры по деньгам.

### 4.6. Деперсонализация ПДн (ключевой механизм)
- Слой инструментов возвращает модели **обезличенные представления**: `booking
  #A1B2`, `категория Студия`, `3 ночи`, `к оплате 18 000 ₽`, `статус: ожидает
  оплаты`. ФИО, номер/скан паспорта, телефон, email в текст, уходящий в модель,
  **не подставляются**.
- Отправка документов/подтверждений и привязка оплаты выполняются backend'ом по
  внутренним ID — модель оперирует ссылкой-идентификатором, а реальные ПДн
  подставляет backend уже после решения модели.
- Тест приёмки: перехват исходящего payload к LLM не содержит паспортных данных,
  ФИО и контактов (кроме случаев, явно разрешённых конфигом).

### 4.7. Эскалация на человека
- Триггеры: низкая уверенность модели, чувствительная тема (деньги/споры/жалобы),
  явный запрос гостя, отмена/возврат.
- Действие: `escalate_to_admin` → диалог с полным контекстом попадает в **ленту
  эскалаций админки (operator inbox)**; статус `AiConversation` → `ESCALATED`;
  уведомление оператору. Оператор отвечает прямо из админки — ответ уходит гостю
  в тот же канал (web/app/Telegram). Всё внутри нашего ПО, без внешней CRM.

### 4.8. Делегирование гостевого диалога
- Оператор может **передать диалог гостя другому сотруднику/отделу** (оплата →
  бухгалтер, инженерия → техник) с комментарием-контекстом.
- Диалог сохраняет всю историю; у нового ответственного появляется в его ленте,
  гость продолжает в том же канале без разрыва.
- `EscalationTicket.operatorId` переназначается; лог передач (кто→кому→когда,
  причина) — для аудита и метрик (§5.7).

### 4.9. Контекст и история
- **Контекст держится** на большом числе сообщений: окно 1M токенов (Sonnet/Opus)
  + prompt caching стабильного префикса + server-side compaction для очень
  длинных диалогов (API сам сжимает раннюю историю в резюме).
- **История сохраняется в нашей БД** (Postgres): API модели stateless — при каждом
  запросе историю формирует backend из `AiMessage`. Нужна также для аудита,
  разбора спорных случаев и ПДн-журнала.

## 5. Общие компоненты

### 5.1. `LlmPort` (порт LLM-провайдера)
Интерфейс (по образцу `Bitrix24Port`):
```
complete({ system, messages, tools, model, cache })   → { text, toolCalls, usage }
stream(...)                                            → async iterable
```
Адаптеры: **`DeepSeekAdapter`** (OpenAI-совместимый API DeepSeek — основной;
Qwen/GLM подключаются тем же OpenAI-совместимым адаптером сменой endpoint+ключа),
`YandexGptAdapter` (managed, РФ), `ClaudeAdapter` (Anthropic SDK), `MockLlmAdapter`
(тесты, без сети — как `mock-bitrix24.adapter.ts`). Выбор — `LLM_PROVIDER` в
конфиге.

### 5.2. `EmbeddingPort`
`embed(texts) → vectors`. Адаптеры: Voyage/OpenAI/Yandex embeddings.
Для базы знаний (без ПДн) провайдер не критичен. Требует pgvector (§6).

### 5.3. Реестр инструментов
Единый реестр `tools/*` с метаданными: имя, JSON-schema входа, требуемое право
(для копилота), уровень «read/write/prepare», обёртка деперсонализации (для
гостевого агента). Реестр фильтруется по контексту (права сотрудника / гость)
перед каждым запросом к модели. Строгая валидация входов (`strict: true`).

### 5.4. RAG-сервис
Индексация документов базы знаний, векторный поиск top-k, сборка контекста с
цитатами (источник → документ), reindex при изменении документа.

### 5.5. Журнал и метрики
`AiToolAudit` (все вызовы инструментов), `AiMessage.usage` (токены), дашборд
стоимости по агентам/каналам/моделям. Метрики: доля автоответов, доля эскалаций,
конверсия «диалог → бронь», средняя стоимость диалога.

### 5.6. Стиль общения (тон агентов)

Зафиксировано 2026-07-08. Тон задаётся системным промптом каждого агента.

**Гостевой агент — «Тёплый дружелюбный»:**
- обращение на «Вы», доброжелательно, с искренней заботой; лёгкие эмодзи умеренно
  (0–1 на сообщение), уместные для гостеприимства;
- в проблемных ситуациях начинать с поддержки («не переживайте», «сейчас
  разберёмся»); короткие абзацы, живой язык, без канцелярита;
- всегда предлагать следующий шаг; цены/доступность — только из инструментов, не
  выдумывать; чувствительные темы / деньги / споры — мягко подключить
  администратора; паспорт/полные данные в чат не запрашивать — направлять в
  защищённую форму (§8).
- Эталон: «Здравствуйте! Отличный выбор 🙂 На эти выходные есть свободная Студия —
  за две ночи выйдет 18 000 ₽ вместе с завтраком. Хотите, я всё подготовлю и
  пришлю ссылку на оплату?»

**Копилот сотрудника — «Дружелюбный коллега»:**
- тон помощника-коллеги, обращение на «ты» допустимо; по-человечески, но без
  лишних слов; проявлять понимание ситуации;
- сразу давать варианты и предлагать действие; перед write-действием — предложить
  и дождаться подтверждения; опираться на данные инструментов, не выдумывать.
- Эталон: «Понимаю, неприятная ситуация. Давай так: можно предупредить соседей —
  оформлю задачу администратору — или предложить гостю из 305 переехать, сейчас
  свободна 312. Что выбираем? Скажи — и я всё оформлю.»

**Общее для обоих:** русский язык; не выдумывать факты (цены, доступность,
правила — только из инструментов/базы знаний); при незнании — честно сказать или
эскалировать; соблюдать RBAC и подтверждения действий; соблюдать правила ПДн (§8).

### 5.7. AI-аналитика и контроль качества чатов

AI-агент анализирует переписку администраторов с гостями (и при необходимости
внутренние чаты) на соответствие стандартам работы и общения, считает метрики,
формирует дашборд и алерты.

**Оценка качества (QA-скоринг):**
- Чек-лист стандартов: приветствие, вежливость и тон, полнота ответа, следование
  скриптам/регламентам, предложен ли апселл, вовремя ли эскалация, отсутствие
  грубости/игнора, корректность по ПДн.
- По каждому диалогу — балл + разбивка по критериям + авто-флаги нарушений
  (грубость, игнорирование, долгий ответ, обещание без выполнения).
- Разбор батчево (Batch API — ~2× дешевле) на **обезличенных/маскированных**
  транскриптах (§8).

**Метрики (по оператору / объекту / периоду):**
- **скорость первого ответа** (first response time);
- **время подключения к чату** (time-to-pickup — от эскалации до ответа оператора);
- среднее время ответа, время решения (resolution time);
- доля диалогов в SLA, доля эскалаций, доля автоответов AI vs человек;
- тональность/удовлетворённость гостя (sentiment / CSAT-прокси);
- нагрузка (диалогов на оператора), конверсия «диалог → бронь».

**Вывод:** дашборд в админке (право `ai_qa`), периодические отчёты, алерты по
нарушениям стандартов и выходу за SLA. Данные — `QaReview`, `ChatMetric` (§6).

## 8. Безопасность и ПДн (152-ФЗ)

Два разных потока ПДн — закрываются по-разному.

**Поток 1 — данные из бэкенда** (что модель читает через инструменты).
Инструменты возвращают обезличенные представления по ID (`бронь #A1B2`, `Студия`,
`3 ночи`, `18 000 ₽`). Модель оркестрирует по ID, реальные ПДн подставляет backend
уже после решения модели (§4.6). Модели **не нужен** паспорт/ФИО, чтобы оформить
бронь: она решает «что», backend знает «для кого» (по сессии авторизованного
гостя). Скан/номер паспорта, полные ФИО, контакты **никогда** не уходят в модель
без явного разрешения конфигом.

**Поток 2 — входящий текст гостя** (что гость сам пишет). Он может содержать ПДн
(«Меня зовут …, тел. …») и попадает в контекст модели — деперсонализация
инструментов его не покрывает. Меры:
- **Маскирование на входе:** перед отправкой в модель — редакция (regex на
  телефон/email/серию-номер паспорта + NER на ФИО) → плейсхолдеры `[ИМЯ]`,
  `[ТЕЛЕФОН]`; соответствие держит backend.
- **Структурный сбор:** идентификация/контакты/паспорт — через **защищённую
  форму** (app/web), данные идут прямо в backend, минуя модель.
- **Резидентность:** при провайдере **YandexGPT (РФ)** трансграничной передачи
  нет вовсе — обработка входящего текста остаётся в Yandex Cloud (РФ). Это делает
  RU-модель юридически самым простым вариантом именно из-за потока 2.

**Юридический контур:**
- **Локализация (ст. 18 152-ФЗ):** первичная запись ПДн граждан РФ — в БД на
  территории РФ. У нас backend + Postgres в Yandex Cloud (РФ) ✓.
- **Трансграничная передача (ст. 12):** отправка ПДн в Claude (США) — передача в
  страну без «адекватной защиты» → требует **явного согласия** на трансграничную
  передачу + уведомления Роскомнадзора. При YandexGPT этот пункт не возникает.
- **Согласия:** на обработку ПДн, на общение с AI и (для Claude) на трансграничную
  передачу — в онлайн-регистрации и первом контакте.
- Хранение ПДн — по регламенту (шифрование, доступ, срок хранения, журнал
  просмотров — как в CLAUDE.md).

**Вывод:** деперсонализация (поток 1) не покрывает то, что гость сам напишет
(поток 2). Для варианта Claude обязательны маскирование на входе + структурный
сбор + согласие на трансграничную передачу; вариант YandexGPT снимает
трансграничный вопрос полностью. Финальный выбор — §13.

**Действующее решение (2026-07-08): китайское облако (Qwen/DeepSeek).** Данные
уходят в КНР/Сингапур → трансграничная передача применяется так же, как к Claude:
**обязательны** маскирование входящего текста + структурный сбор паспорта через
форму + явное согласие гостя на трансграничную передачу + уведомление РКН.
Локализация (ст. 18) соблюдена — первичная запись в нашем Postgres в РФ.
Рекомендуется международный регион провайдера (напр. Alibaba Cloud Singapore),
чтобы данные шли не в материковый Китай.

- Все вызовы инструментов, читающих данные гостя, логируются (`AiToolAudit`).
- HTTPS, JWT, ключи LLM/эмбеддингов — только на backend.

## 11. Критерии приёмки

- Овербукинг невозможен: бронь агента идёт через `pms/bookings` (PG-транзакция +
  `Idempotency-Key`); нагрузочный тест на гонки — двойной брони нет.
- Деперсонализация: перехват payload к LLM не содержит паспорт/ФИО/контакты.
- RBAC-гейтинг: копилот не вызывает инструмент вне прав роли (тест на front-desk
  vs складской инструмент).
- «Готовит — гость подтверждает»: агент не списывает деньги без оплаты гостем;
  отмена/возврат уходит человеку.
- Эскалация: диалог с полным контекстом попадает в открытую линию Bitrix24,
  ответ сотрудника возвращается гостю.
- История сохраняется в БД; контекст держится на длинном диалоге (тест на
  compaction).
- Стоимость и токены логируются по каждому диалогу.
- Тип объекта: привилегии раннего заезда/позднего выезда/апгрейда доступны только
  для отелей, не для апартаментов.

Flash $1.50/$9 · Gemini 3.1 Pro $2/$12.

### 14.1. Китайские модели и «бесплатные» варианты (июль 2026)

Китайские LLM сильны и дёшевы; ключевая особенность — большинство **open-weight**
(можно self-host, без поштучной платы за токены).

Хостинг-API (за 1М токенов, вход/выход): DeepSeek V4 Flash $0.14/$0.28 · V4 Pro
$0.44/$0.87 · R1 $0.55/$2.19 · Qwen (open) от ~$0.15 вход · Qwen 3.7-Max $2.50
вход · Kimi K2.6 $0.60/$2.50 · GLM-5.1 (Zhipu) — через провайдеров. Дешевле
Claude/GPT/Gemini в разы.

**«Бесплатные» — в двух смыслах:**
- **Open-weight (Apache/MIT) → бесплатный self-host:** Qwen (Apache 2.0),
  DeepSeek, Kimi K2, GLM — веса открыты; на своём GPU платите только за сервер, не
  за токены. **Развёрнутые на нашем Yandex Cloud GPU, они дают data-residency (ПДн
  в РФ) + отсутствие платы за токены** — закрывают сразу стоимость и 152-ФЗ для
  гостевого агента. Минус: GPU-сервер + эксплуатация; качество ниже Claude-frontier
  (но Qwen/DeepSeek/GLM сильные).
- **Пробные бесплатные кредиты хостинг-API:** DeepSeek — 5M токенов при регистрации
  (30 дней); Qwen Model Studio — ~70M токенов на 90 дней. Это триал, не постоянно
  бесплатно.

**Важно:** обращение к **китайскому хостинг-API** = данные уходят в КНР → та же
(или большая) проблема трансграничной передачи по 152-ФЗ + геополитические/
безопасностные риски. Поэтому из китайских вариантов для гостевого агента с ПДн
интересен **только self-host open-weight на нашей инфраструктуре** (данные не
покидают РФ). Для копилота (мало ПДн) допустим и хостинг-API, но экономия
относительно Haiku невелика.

**Итог по вариантам гостевого агента** (все три — за одним `LlmPort`):
1. **Claude + маскирование** — лучшее качество (managed, но трансграничная
   передача → согласие + РКН);
2. **YandexGPT** — просто по ПДн, managed, данные в РФ;
3. **Self-host open-weight (Qwen/DeepSeek/GLM) на Yandex Cloud** — ПДн в РФ + без
   платы за токены, ценой GPU-сервера и эксплуатации.

**Выбор 2026-07-08 — вариант 1 (китайское облако / hosted API), провайдер
`DeepSeek`, НЕ self-host.** Модели: **V4 Flash** ($0.14/$0.28 за 1М — одна из самых
дешёвых LLM) для роутинга/простого, **V4 Pro** ($0.44/$0.87) для основной работы,
**R1** для reasoning-задач; надёжный function-calling, OpenAI-совместимый API,
дешёвое чтение кэша. Плюсы облака: ноль инфраструктуры, оплата по факту, масштаб;
для умеренного объёма отеля дешевле выделенной GPU. **Минус (важно):** endpoint в
материковом КНР → данные уходят за рубеж → трансграничная передача по 152-ФЗ
(согласие + РКН) + геополитические/доступностные риски → маскирование входящего
текста (§8) обязательно. Эмбеддинги для RAG — DeepSeek/Qwen embeddings API или
открытые `bge-m3`/`multilingual-e5` (self-host, дёшево). Провайдеры взаимозаменяемы
через OpenAI-совместимый адаптер за `LlmPort` (Qwen/Alibaba, YandexGPT, Claude — как
резерв / A-B).
