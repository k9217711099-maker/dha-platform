## 1. Архитектурные принципы (не нарушать)

1. **Всё в монорепо**: backend — `apps/api/src/kb/*` и `apps/api/src/drive/*`
   (NestJS + Prisma + Postgres), UI — разделы «База знаний» и «Диск» в админке
   `apps/admin` (Next.js, тема «Спокойный индиго», Manrope).
2. **Мультиарендность**: все сущности с `tenantId`, как в PMS (Путь B).
3. **Файлы — только S3-совместимое хранилище** (Yandex Object Storage;
   локально — встроенный fallback на диск за общим `StoragePort`). В Postgres —
   только метаданные. Прямой отдачи S3-ключей клиенту нет: скачивание через
   backend (проверка прав) либо короткоживущие presigned URL.
4. **Поиск учитывает права** (ACL-aware): фильтрация по доступам выполняется
   **до** формирования выдачи и до передачи фрагментов в LLM. Пользователь не
   должен увидеть даже сниппет документа, к которому у него нет доступа. Это
   же правило действует для RAG копилота из `AI-COMMUNICATIONS-TZ.md` §3.4.
5. **LLM и эмбеддинги — только через порты** `LlmPort` / `EmbeddingPort`
   (уже в `apps/api/src/ai/`); провайдер — DeepSeek (решение владельца),
   сменяемый конфигом. Перед отправкой фрагментов в облачную модель —
   маскирование ПДн (`PiiMaskingService`).
6. **Единый поисковый контур** для БЗ и Диска: одна таблица чанков, один
   индексатор, один API поиска — источники различаются типом ресурса.
7. **Аудит**: просмотр/изменение/выдача прав/создание публичных ссылок/скачивание
   логируются (кто, что, когда, откуда) — общий журнал действий админки.

Схема:

```
Админка (Next.js)                        Мобайл/гость
  ├─ Раздел «База знаний»                  └─ (v2: публичные страницы БЗ,
  └─ Раздел «Диск»                              напр. правила проживания)
        │ REST
        ▼
apps/api (NestJS)
  ├─ kb/*        — базы, страницы, версии, импорт B24
  ├─ drive/*     — папки, файлы, версии, WOPI-хост, карты
  ├─ search/*    — индексатор + гибридный поиск + AI-ответы
  ├─ acl/*       — общий сервис прав (ACL поверх RBAC)
  └─ ai/*        — LlmPort/EmbeddingPort/PII (уже есть)
        │
        ├─ Postgres (+ pgvector) — метаданные, тексты, эмбеддинги
        ├─ S3 (Yandex Object Storage) — файлы, ассеты страниц
        └─ Collabora Online (Docker, отдельный контейнер) — WOPI-редактор
```

## 2. Модель доступа (общая для БЗ и Диска)

Два уровня, работают вместе:

1. **RBAC (глобальные права роли)** — в `admin/permissions.ts`, определяет,
   какие разделы и операции в принципе доступны роли (см. §10).
2. **ACL (права на объект)** — гранты на конкретную базу знаний / раздел /
   страницу / папку / файл. Субъект гранта: пользователь, роль или **группа**
   (новая сущность `UserGroup`, например «Ресепшен Босфор», «Горничные»).

Уровни доступа (единая шкала для обоих модулей):

| Уровень | БЗ | Диск |
|---|---|---|
| `viewer` | читать | смотреть/скачивать |
| `commenter` (v2) | комментировать | комментировать |
| `editor` | редактировать страницы | загружать/менять файлы |
| `manager` | + структура, права, публичные ссылки | + права, публичные ссылки, корзина |

Правила:
- **Наследование сверху вниз** (база → раздел → страница; папка → подпапка →
  файл) с возможностью переопределить на любом узле (расширить или ограничить;
  ограничение — флаг «не наследовать» на узле).
- У каждого объекта есть владелец (создатель) — всегда `manager` на свой объект.
- `superadmin` видит всё; кнопка «Открыть как…» для проверки, что видит роль.
- UI: страница «Доступы» на любом узле — список грантов, наследуемые показаны
  серым с указанием источника («унаследовано от: База „Регламенты СПиР“»).

## 3. База знаний — структура и редактор

### 3.1. Структура

- **Базы знаний** (несколько на тенанта: «Регламенты», «СПиР», «УК», …) →
  внутри **дерево страниц** произвольной вложенности (страница может быть и
  разделом — как в Notion/Outline, без искусственного деления «раздел/статья»).
- У страницы: заголовок, иконка (эмодзи), контент (JSON-документ), автор,
  дата изменения, статус (`draft`/`published`/`archived`), теги.
- Сайдбар с деревом, drag-and-drop перемещение (с проверкой прав), избранное,
  «Недавние».

### 3.2. Редактор

- Блочный WYSIWYG на **TipTap (ProseMirror)** — хранение в JSON.
- Блоки MVP: заголовки H1–H3, текст, списки (вкл. чекбоксы), таблица, цитата,
  callout (инфо/предупреждение), код, разделитель, изображение, видео (файл/
  ссылка), файл-вложение (лежит на Диске, см. §5.1), эмбед ментальной карты.
- Внутренние ссылки: `@упоминание` сотрудника и `[[ссылка]]` на другую страницу
  БЗ / файл Диска с автодополнением; битые ссылки подсвечиваются.
- **Версии**: снапшот при каждой публикации + автоснапшот не чаще раза в
  10 минут при редактировании; история версий с diff и откатом.
- Одновременное редактирование: MVP — мягкая блокировка («страницу редактирует
  Анна, открыть только для чтения / перехватить»); realtime co-editing (Yjs) — v2.

### 3.3. Ссылки на страницы (требование владельца)

- Каждая страница получает **постоянный адрес** вида
  `/kb/{base-slug}/{page-slug}-{shortId}`, где `shortId` — неизменяемый
  8-символьный код. Переименование меняет slug, но старые ссылки продолжают
  работать (резолв по `shortId` + 301 на актуальный адрес).
- Кнопка «Скопировать ссылку» у страницы и **у каждого заголовка** (якоря
  `#h-<anchor>`); ссылка на конкретную версию — `?v=<n>`.
- Внутренняя ссылка открывается только авторизованным с правом `viewer`+.
- **Публичная ссылка** (для гостей/партнёров, вне логина): включается на
  страницу явно правом `manager`, общий механизм `PublicLink` — см. §5.4.

### 3.4. Импорт из Bitrix24 (ZIP)

Экспорт БЗ в Bitrix24 отдаёт ZIP-архив: папки + JSON-описания страниц
(база знаний B24 построена на конструкторе «Сайтов» — страницы состоят из
блоков) + файлы-ассеты (картинки, документы).

Пайплайн импорта (`kb/import/bitrix24`):

1. **Загрузка ZIP** в админке (мастер импорта), файл — во временное S3.
2. **Разбор**: распаковка, чтение JSON-манифестов; построение дерева
   «база → разделы → страницы»; извлечение блоков и ассетов.
3. **Dry-run предпросмотр** (обязательный шаг): сколько страниц/картинок/файлов
   найдено, дерево будущей структуры, список нераспознанных блоков — до
   какой-либо записи в БД.
4. **Маппинг блоков** B24 → наш JSON: текст/заголовки/списки/картинки/файлы/
   видео/колонки → соответствующие блоки TipTap; колонки схлопываются в
   последовательность. **Нераспознанный блок не теряется**: его HTML
   сохраняется в raw-HTML-блок с пометкой «требует ручной доработки»,
   страница попадает в список «на доработку».
5. **Ассеты** → S3, ссылки в контенте переписываются; внутренние ссылки между
   страницами B24 конвертируются в наши `[[ссылки]]` (по таблице соответствия
   URL → страница).
6. **Идемпотентность**: у импортированных страниц хранится `externalId`
   (идентификатор из выгрузки); повторный импорт того же архива — режимы
   «пропустить существующие» / «обновить» (по выбору), дублей не создаёт.
7. **Отчёт**: создано / обновлено / пропущено / на доработку, лог ошибок.
   Импорт — фоновая задача (`ImportJob`) с прогрессом; большие архивы (сотни
   страниц, гигабайты ассетов) не блокируют API.
8. После импорта автоматически запускается индексация для поиска (§4).

Ограничение (честно): формат выгрузки B24 не задокументирован публично и может
отличаться между тарифами/версиями. Первый шаг реализации — **получить реальный
ZIP с портала владельца** и зафиксировать фактическую структуру; маппер пишется
по реальному образцу, а не по догадкам. Это блокер этапа — см. §13.

### 3.5. Быстрое внесение информации через AI-агента

Работает внутри **копилота сотрудника** (`AI-COMMUNICATIONS-TZ.md` §3) — новые
инструменты `kb_search`, `kb_draft_page`, `kb_append_page`:

- Сценарии: «сохрани в базу знаний: …» (текст/диктовка), пересланный файл или
  фрагмент переписки, «оформи из этого регламент».
- Агент: (1) ищет семантически похожие страницы; (2) предлагает — **дополнить
  существующую** страницу или **создать новую** (и в каком разделе);
  (3) оформляет текст в структуру (заголовки, списки, callout) в тоне
  существующих регламентов; (4) создаёт **черновик**.
- **Публикует всегда человек**: агент не имеет права публиковать и не может
  менять опубликованный текст без подтверждения (принцип «готовит — сотрудник
  подтверждает», как у гостевого агента). Diff «что добавится» показывается
  перед подтверждением.
- Права: инструменты выполняются от имени сотрудника — куда у него нет
  `editor`, туда агент писать не может.
- Атрибуция: у страницы/версии помечается «создано с помощью AI, подтвердил
  <сотрудник>».

## 4. Интеллектуальный поиск (общий для БЗ и Диска)

Требование «поиск с помощью AI» + «интеллектуальный поиск» в обоих модулях —
реализуется одним контуром `search/*`.

### 4.1. Индексация

- Единая таблица чанков `SearchChunk` (тип ресурса: `kb_page` | `drive_file`,
  id ресурса, № чанка, текст, `tsvector`, `embedding vector`).
- БЗ: страница → чанки по заголовкам/абзацам (300–500 токенов, overlap).
- Диск: **извлечение текста из файлов** — `docx`/`xlsx`/`pptx` (mammoth / xlsx /
  разбор XML), `pdf` (pdf-parse; сканы без текстового слоя — OCR в v2),
  `txt`/`md`/`csv`, ментальные карты (текст узлов). Прочие форматы — только
  имя/метаданные.
- Индексатор — фоновая очередь на БД + `@nestjs/schedule` (как синк каналов),
  триггеры: публикация/изменение страницы, загрузка/новая версия файла, импорт.
  Удаление ресурса → удаление чанков.
- Эмбеддинги через `EmbeddingPort`; pgvector во встроенном Postgres — проверить
  доступность расширения, fallback MVP: только FTS, эмбеддинги при деплое
  на сервер (см. §13).

### 4.2. Гибридный поиск

- **Полнотекстовый** (Postgres FTS, конфигурация `russian`) + **векторный**
  (pgvector, cosine) параллельно → слияние **RRF** (reciprocal rank fusion).
- Фильтры: модуль (БЗ/Диск/везде), база/папка, автор, тип файла, теги, даты.
- **Фильтрация по ACL до выдачи** (принцип §1.4).
- Опечатки/морфология: FTS-словари + триграммы (`pg_trgm`) для «похожих» имён
  файлов; векторная ветка закрывает синонимы и перефразировки («как заселить
  гостя ночью» найдёт «регламент ночного заезда»).
- Единая строка поиска в шапке админки: мгновенные подсказки (заголовки/имена
  файлов) → полная выдача со сниппетами и подсветкой.

### 4.3. AI-ответ (RAG)

- Режим «Спросить базу знаний»: top-K чанков (после ACL-фильтра и
  PII-маскирования) → `LlmPort` → ответ **со ссылками-цитатами** на конкретные
  страницы/файлы (клик — переход на страницу к якорю).
- Если релевантных чанков нет — честное «в базе знаний ответа нет», без
  фантазий; предложение создать страницу (мостик к §3.5).
- Этот же контур используется копилотом и гостевым AI-администратором
  (для гостевого — только страницы, явно помеченные «доступно гостевому
  агенту», отдельный флаг на странице).

## 5. Диск

### 5.1. Хранилище и структура

- Разделы: **Общий диск** (структура компании), **Мой диск** (личный раздел
  сотрудника), **Файлы Базы знаний** (служебный, ассеты страниц — техраздел,
  управляется автоматически).
- Дерево папок неограниченной вложенности; операции: загрузка (drag-and-drop,
  папками, мультизагрузка), создание/переименование/перемещение/копирование,
  список/плитка, сортировка, хлебные крошки.
- **Версии файлов**: загрузка файла с тем же именем в ту же папку → новая
  версия (история, откат, скачивание старой версии). Хранение версий — по
  настройке (по умолчанию 10 последних или 90 дней).
- **Корзина**: мягкое удаление, восстановление, автоочистка через 30 дней;
  «удалить навсегда» — только `manager`.
- Предпросмотр: изображения, PDF, видео/аудио (стрим), office-форматы — через
  Collabora в режиме read-only (или конвертация в PDF для превью).
- Квоты на тенанта (SaaS-план) и опционально на «Мой диск»; экран «занято
  места».
- Метаданные: размер, mime, чек-сумма (sha256, для дедупликации хранения —
  одинаковые блобы храним один раз), кто загрузил, когда.

### 5.2. Онлайн-редактирование Microsoft Office

Решение: **WOPI-протокол + Collabora Online** (self-hosted, Docker-контейнер
рядом с API).

- Почему Collabora, а не альтернативы:
  - настоящий Microsoft Office for the web лицензионно доступен только через
    Microsoft 365/облако Microsoft — для self-hosted платформы не вариант;
  - ONLYOFFICE Docs Community ограничен **20 одновременными подключениями**
    и режет мобильные браузеры в бесплатной версии;
  - Collabora Online (CODE/community) — MPL 2.0, **без лимита пользователей**,
    полноценный WOPI-хостинг, совместное редактирование.
  - Интерфейс у ONLYOFFICE ближе к MS Office; если владельцу критичен вид «как
    Word» и ок платная лицензия — архитектура не меняется (оба работают по
    WOPI/похожему протоколу), см. §13.
- Наш API реализует **WOPI-хост**: `CheckFileInfo`, `GetFile`, `PutFile`,
  `Lock/Unlock/RefreshLock`; `access_token` — короткоживущий JWT с правами
  конкретного пользователя на конкретный файл (view/edit по ACL).
- Форматы: `docx`, `xlsx`, `pptx` — редактирование; `doc/xls/ppt` — открытие с
  конвертацией; `odt/ods/odp` — нативно.
- **Совместное редактирование** нескольких сотрудников в одном документе —
  из коробки Collabora (курсоры, presence).
- Создание документа «с нуля» из Диска: «Создать → Документ/Таблица/Презентация».
- Сохранение `PutFile` создаёт новую версию файла (§5.1).
- Инфраструктурное требование: Docker на сервере (в проде — Yandex Cloud VM,
  уже в планах инфры; в локальном dev-окружении Docker нет → в dev
  онлайн-редактирование выключено фиче-флагом, файлы скачиваются/загружаются).

### 5.3. Роли и доступы

- Общая модель §2: ACL на папку/файл, наследование, гранты пользователям/
  ролям/группам, уровни `viewer`/`editor`/`manager`.
- «Мой диск» приватен по умолчанию; расшаривание конкретных папок/файлов —
  внутренней ссылкой с грантом («поделиться с сотрудником/группой»).
- Типовые сценарии: бухгалтерия видит «Финансы», СПиР — «Регламенты СПиР»,
  фотографии номеров доступны маркетингу на `editor`, всем остальным `viewer`.

### 5.4. Внутренние и публичные ссылки (общий механизм `PublicLink`)

- **Внутренняя ссылка**: постоянный URL `/drive/f/{shortId}` (файл) и
  `/drive/d/{shortId}` (папка); открывается только авторизованным с правом
  `viewer`+; кнопка «Скопировать ссылку» везде.
- **Публичная ссылка** (файл, папка, страница БЗ — единая сущность):
  - создаётся явно правом `manager`; токен ≥ 128 бит энтропии
    (`/s/{token}`), без перебора;
  - опции: срок действия, пароль, право (только просмотр / просмотр+скачивание;
    для папок — просмотр списка), запрет индексации (`noindex`, `X-Robots-Tag`);
  - отзыв в один клик; список всех активных публичных ссылок тенанта — экран
    в админке (аудит «что у нас торчит наружу»);
  - счётчик открытий + журнал (время, IP) в аудите;
  - публичная страница отдаётся в лёгком «гостевом» лейауте без админки.

### 5.5. Ментальные карты (как XMind)

- Тип файла на Диске (`.dmap`, JSON) — карта живёт в общей структуре папок,
  наследует ACL, версии, ссылки, поиск (текст узлов индексируется §4.1).
- Редактор в админке на **mind-elixir** (MIT, зрелая JS-библиотека,
  framework-agnostic): узлы/подузлы (Tab/Enter как в XMind), drag-and-drop
  ветвей, сворачивание, цвета/иконки узлов, заметка и ссылка на узле
  (в т.ч. `[[ссылка]]` на страницу БЗ), связи между ветвями, undo/redo.
- Экспорт: PNG, SVG; импорт `.xmind` — v2 (это ZIP с JSON-контентом, реально);
  экспорт в `.xmind` — v2.
- Совместное редактирование: MVP — мягкая блокировка как в БЗ §3.2; realtime — v2.
- Встраивание карты в страницу БЗ (блок-эмбед, интерактивный просмотр).

## 8. Пароли и секреты — модуль «Секреты» (решение владельца 2026-07-08)

**Пароли не хранятся в страницах Базы знаний открытым текстом — запрещено.**
Причины: страницы попадают в поисковый индекс, в эмбеддинги (уходят во внешнюю
LLM), в версии, в публичные ссылки; отследить, кто видел пароль, невозможно.

**Решено: строим свой модуль «Секреты»** (Vaultwarden/Passwork отклонены —
не интегрируются с нашим офбордингом без доп. работ):

- Отдельное шифрованное хранилище: AES-256-GCM, ключ шифрования вне БД
  (env/KMS), пароль раскрывается по клику «Показать» с записью в журнал.
- Карточка секрета: название, логин, пароль/ключ, URL кабинета, ответственный
  за ротацию, комментарий, теги; ACL по общей модели §2.
- **Журнал каждого просмотра** (кто, какой секрет, когда) — основа офбординга.
- **Офбординг-процедура**: при отключении сотрудника система строит список
  секретов, к которым он имел доступ / которые просматривал → авто-создаются
  задачи на ротацию ответственным, дашборд «ротация после увольнения N из M»,
  напоминания до полного закрытия. После смены пароль обновляется в карточке —
  задача закрывается.
- Авто-ротация «кнопкой» — только для систем с API смены пароля (наша
  платформа, наши сервисы). Для внешних кабинетов без API — workflow задач
  (см. пояснение ниже).
- Детектор секретов в БЗ: при сохранении страницы regex-эвристики («пароль:»,
  «pwd», ключи API) → предупреждение «перенесите в модуль Секреты» + кнопка
  переноса.

**Пояснение про внешние кабинеты (Островок, эквайринг и т.п.).** Готовых
надёжных решений авто-смены пароля в чужих веб-кабинетах нет: такие функции
есть только в enterprise-PAM (CyberArk и аналоги) через сценарии
роботизированного браузера — дорого, ломается при любом изменении вёрстки и
не работает при SMS-подтверждении/капче, для наших сервисов непрактично.
Правильный путь двойной: (1) **личные аккаунты вместо общего пароля** — и
экстранет Островка, и кабинеты ЮKassa/банков поддерживают добавление
сотрудников со своими логинами и ролями; тогда увольнение = удаление его
аккаунта в кабинете (задача офбординга), общий пароль не трогаем;
(2) для кабинетов, где личных аккаунтов нет, — смена вручную по авто-задаче
из офбординга: задача открывается, пароль меняется в кабинете, новое значение
вносится в карточку секрета, задача закрывается. Автоматизирована гарантия
«ничего не забыли», а не сам ввод пароля.

**Главная мера — организационная: минимум общих паролей.** Каждому сотруднику —
личный аккаунт во всех системах, где возможно; тогда увольнение = отключение
аккаунта (наша админка это уже умеет), и ротация общих секретов нужна лишь для
короткого списка внешних кабинетов.

## 9. Безопасность и ПДн

- Всё за авторизацией админки (JWT), кроме `/s/:token` (публичные ссылки).
- Публичные ссылки: токен ≥128 бит, rate-limit, опц. пароль/срок, `noindex`.
- Скачивание файлов — только через backend/presigned URL с TTL ≤ 5 мин.
- WOPI: `access_token` короткоживущий, привязан к файлу и пользователю;
  Collabora ходит только в наш API, наружу не торчит (внутренняя сеть).
- В LLM/эмбеддинги фрагменты уходят **после** `PiiMaskingService` (тексты БЗ
  могут содержать телефоны/ФИО в примерах) — трансграничная передача, 152-ФЗ
  (см. AI-ТЗ §8).
- Сканы документов гостей на Диске не хранятся — для них отдельный защищённый
  контур из основного ТЗ (`checkin`); Диск — для рабочих файлов компании.
- Антивирус-проверка загрузок (ClamAV) — v2; MVP — ограничение опасных
  расширений для публичных ссылок.
- Лимиты: размер файла (по умолчанию 2 ГБ), размер импорт-ZIP, rate-limit на
  загрузку.

## 12. Критерии приёмки (MVP)

1. ZIP с реального портала B24 владельца импортируется: ≥95% страниц читаемы
   без ручной правки, все ассеты на месте, отчёт показывает страницы
   «на доработку»; повторный импорт не создаёт дублей.
2. Поиск «как заселить гостя ночью» находит страницу «Регламент ночного
   заезда» (синонимы/перефраз), выдача не содержит объектов без доступа
   (проверка под ограниченной ролью).
3. «Спросить базу знаний» отвечает с цитатами-ссылками; на вопрос вне базы
   отвечает «в базе знаний этого нет».
4. Ссылка на страницу/заголовок работает после переименования страницы.
5. Публичная ссылка: открывается без логина, истекает по сроку, отзывается,
   фиксируется в аудите; закрытая — не открывается.
6. ACL: сотрудник без гранта не видит базу/папку ни в дереве, ни в поиске,
   ни по прямой ссылке; наследование и переопределение работают.
7. Версии страницы и файла: история, diff (для страниц), откат.
8. Файл 500 МБ загружается и скачивается; корзина восстанавливает; квота
   считается.
9. (v1.5) docx открывается в браузере, двое редактируют одновременно,
   сохранение создаёт версию; карта: создать/ветвить/раскрасить/экспорт PNG.
