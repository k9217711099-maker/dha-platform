## 2. Ошибки и риски в коде (по приоритету)

### P0 — закрыть до боевого запуска

1. **Webhook платежей не верифицируется** (`payments.service.ts → handleWebhook`).
   Статус берётся из тела запроса и применяется напрямую (`applyStatus`). Публичный
   `POST /api/payments/webhook` позволяет злоумышленнику, узнавшему `gatewayPaymentId`,
   пометить бронь оплаченной.
   **Рекомендация:** при получении webhook не доверять телу — перечитывать статус через
   `gateway.getPayment()` (фолбэк-поллинг уже так делает), плюс IP-allowlist диапазонов YooKassa.

2. **Нет миграций Prisma** — схема (131 модель) накатывается только `db push`.
   Для боевой БД это неуправляемо: нет истории изменений, нет отката, риск потери данных
   при расхождении схем. **Рекомендация:** зафиксировать baseline-миграцию
   (`prisma migrate dev --name init`) до переезда на managed PostgreSQL и дальше жить на миграциях.

3. **Нет rate-limiting нигде** (ThrottlerModule отсутствует). Особенно критично:
   - `POST /auth/otp` — нет cooldown на отправку кода (`otp.service.ts:request`):
     при включении боевого SMSC это открытый SMS-флуд (расход бюджета + спам людям).
     Проверка кода защищена (MAX_ATTEMPTS=5), а вот отправка — нет.
   - публичный гостевой AI-эндпоинт (свой TODO в `guest-agent.controller.ts:17`) — расход токенов LLM;
   - публичные ссылки `/s/:token` (КБ/Диск) — перебор токенов.
   **Рекомендация:** `@nestjs/throttler` глобально + отдельные строгие лимиты на OTP
   (не чаще 1 SMS в 60 сек на номер, суточный потолок) и AI-чат.

### P1 — желательно до запуска

4. **2FA администраторов не реализована** — прямое требование ТЗ §18.1
   (TODO в `admin-auth.service.ts:17`). Админка управляет ключами замков и ПДн.
5. **Bitrix24 webhook без проверки подписи** (`chat.controller.ts:29`, свой TODO).
   Модуль легаси, но эндпоинт публичный — либо закрыть фичефлагом, либо добавить секрет.
6. **Компенсация при сбое создания брони в Bnovo-контуре** (`booking.service.ts:76`, свой TODO):
   при падении БД после создания брони во внешней PMS бронь-сирота остаётся. Контур легаси
   (ядро уже на своём PMS), но код жив и доступен с фронта.
7. **`/uploads` отдаётся статикой без авторизации** (`main.ts:21`). Имена файлов —
   `randomUUID` (неперебираемо), но вложения задач, фото и документы КБ доступны всем,
   у кого есть URL, навсегда. Для сканов паспортов это НЕ используется (там шифрование +
   StoragePort — корректно). **Рекомендация:** при переходе на S3 — приватный бакет +
   подписанные URL с TTL; локально — auth-gate на раздачу.
8. **Ingestion-канал OTA с необязательным токеном** (`channel-ingestion.controller.ts` —
   `X-Channel-Token` required: false; если в credentials канала токен не задан, приём броней
   открыт). Сейчас адаптеры mock; при подключении реальных OTA сделать токен обязательным.
9. **Нет security-заголовков** (helmet) и **нет глобального exception-alerting** —
   ошибки видны только в pino-логах.

### P2 — техдолг, не блокирует

10. Секьюрити-хардening TTLock-пульта: удалённое открытие двери из админки логируется,
    но не требует повторного подтверждения (пин/пароль) — для 200 номеров стоит добавить.
11. КБ-поиск: на локальной среде FTS не токенизирует кириллицу (работает фолбэк);
    GIN-индексы и русский конфиг — на проде (уже запланировано).
12. Известные мелкие пробелы КБ/Диска (из приёмки §12): diff версий в UI, квоты Диска,
    срок/пароль публичной ссылки в UI, мягкие блокировки редактирования, move-UI.
13. Легаси-слой Bnovo (каталог-синк, старый BookingService) жив параллельно новому PMS —
    источник путаницы; после миграции гостевого фронта на Booking Engine — вычистить.

## 3. Соответствие стандартам LQA

### Что уже соответствует (сильные стороны)

| LQA-требование | Реализация |
|---|---|
| Нормативы реакции на заявку гостя | SLA-матрица `OpsSlaPolicy` (severity × «заявка гостя» → accept/due), автопроставление сроков |
| Приоритет гостевых заявок | Флаг `guestRequest` в задачах/шаблонах, отдельные SLA (CRITICAL 10/30 мин) |
| Эскалации при просрочке | Автоматизация: напоминания + эскалации по таймауту статуса с условиями (severity/тег/guestOnly), web-push с «срочной» вибрацией |
| Поведенческие стандарты в номере | Чек-лист «Работа в гостевом номере» (стук/представиться/бахилы/убрать за собой) |
| Профилактика (ППР) | `OpsPmRule`: циклы 90 дней, ночная генерация порций, прогноз закрытия очереди |
| Контроль повторных дефектов | Отчёт «Повторные заявки» (номер+тег ≥2 за период) — классический LQA-редфлаг |
| Контроль качества уборки | Статусы DIRTY→CLEAN→INSPECTED, инспекция, аналитика чек-листов, фотоподтверждение |
| Pre-arrival коммуникация | Воронка check-in: invite/напоминания по каналам, magic-link портал, эскалация «гость не готов» |
| Разделение OOS/OOO | Мягкий дефект (продаётся) vs снят с продажи — влияет на пул доступности |
| Аудит действий | AuditLog по мутациям PMS/задач, журнал доступа к документам (152-ФЗ), журнал ключей |

### Чего не хватает до полного LQA-контура

1. **KPI-отчёт по соблюдению SLA** — на дашборде есть «просроченные», но нет
   «% принятых/закрытых в срок», среднего времени реакции по отделам за период.
   Это главный управленческий отчёт LQA-аудита. (Быстро строится: `acceptBy/dueAt`
   и фактические таймстемпы уже в данных.)
2. **Журнал жалоб / service recovery (glitch log)** — жалоба гостя как отдельная
   сущность: причина, компенсация, ответственный, закрытие петли с гостем.
   Сейчас жалоба растворяется в OpsTask.
3. **Профиль предпочтений гостя → операционные "traces"** — LQA требует узнавания
   гостя (подушка, этаж, аллергии) и автозадач под заезд VIP. База гостей есть,
   предпочтений и VIP-механики нет.
4. **Пост-стей опрос / NPS** — reminder на отзыв есть (легаси-CRM стаб), но нет
   собственной анкеты оценки и отчёта по удовлетворённости.
5. **Turndown-сервис** — в типах уборки нет вечернего сервиса (для люкс-сегмента LQA его оценивает).
6. **Lost & Found** — модуля нет (для 50–200 номеров — ежедневная реальность).
7. **Журнал передачи смены (duty log)** — «В смене» есть, структурированного handover нет.
8. **Реестр оборудования (asset registry)** — осознанно отложен в v2; для ППР-глубины понадобится.

## 4. Глазами отеля на 50–200 номеров

### Что уже хорошо ложится на такой масштаб
Ядро PMS выдерживает главные инварианты (анти-овербукинг в PG-транзакции, идемпотентность,
шахматка по сети объектов), RBAC по трём осям, склад, инженерка с ППР, воронка заселения,
канал-менеджер с очередью и dead-letter, 300+ тестов и живые E2E-прогоны.

### Критичные пробелы для этого масштаба

1. **Ночной аудит / закрытие дня** — нет процедуры: фиксация no-show, автопродление/
   закрытие статусов, суточный отчёт (заезды/выезды/выручка/касса). При 100+ номерах
   без этого бухгалтерия и статистика «плывут».
2. **Управленческая отчётность: Occupancy / ADR / RevPAR** — базовая аналитика есть,
   но нет главной тройки метрик отельера ни в каком отчёте (BI отложен осознанно).
3. **Групповые брони и корпоративные договоры** — блоки номеров под группу, room list,
   договорные тарифы юрлиц, безнал с закрывающими. Для 200 номеров группы — норма жизни.
4. **Фолио гостя / разнесение начислений** — вкладка «Счёт» у брони есть, но полноценного
   фолио (пост-чарджи в проживание, разделение плательщиков, депозиты) нет.
5. **Производительность шахматки** — сетка «вся сеть × 30 дней» при 200 номерах = тысячи
   ячеек с бронями; нужен нагрузочный прогон и, вероятно, виртуализация строк.
6. **Реальные OTA-коннекторы** — при 50–200 номерах без живых Ostrovok/Яндекс.Путешествия
   доля OTA обслуживается вручную (сейчас адаптеры mock — по плану, ждут запроса владельца).
7. **Housekeeping-нагрузка**: план уборок есть; не хватает нормативов времени на номер и
   балансировки нагрузки между горничными (сейчас распределение ручное).
8. **Отказоустойчивость**: embedded Postgres, один процесс API, БД-очередь синка —
   для 200 номеров нужен managed PG + бэкапы + мониторинг/алертинг (Sentry/метрики/uptime),
   Redis/BullMQ по мере роста.
9. **Мультиязычность гостевого контура** — только русский; для СПб-потока иностранцев
   портал заселения и AI-агент стоит локализовать (en минимум).

### Что можно улучшить малой кровью (quick wins)
- SLA-KPI отчёт (данные уже есть) — 1–2 дня.
- Отчёт Occupancy/ADR/RevPAR из существующих броней — 1–2 дня.
- Cooldown на OTP + throttler глобально — до 1 дня.
- Перечитывание статуса платежа при webhook — часы.
- Тип уборки TURNDOWN + шаблон — часы.
- Baseline-миграция Prisma — до 1 дня.

## 5. Очередь «после размещения на сервере»

То, что осознанно отложено и активируется/делается после переезда на боевой сервер
(Yandex Cloud, HTTPS, домены):

### Включается конфигурацией (код готов, ждёт прод-среды)
1. **HTTPS-зависимое:**
   - Web Push сотрудникам на iPhone (PWA «На экран Домой» — требует HTTPS);
   - Telegram-бот: `setWebhook` на публичный URL (токен уже вводится из админки);
   - боевой webhook YooKassa в кабинете магазина (фолбэк-поллинг уже работает);
   - `GUEST_PORTAL_BASE_URL` — боевые magic-link ссылки заселения.
2. **Переключение провайдеров (env):**
   - `SMS_PROVIDER=smsc` — реальные SMS (OTP, invite заселения; сейчас dev-лог);
   - `STORAGE_PROVIDER=s3` — Yandex Object Storage (сканы документов; локально mock);
   - `TTLOCK_PROVIDER=http` — боевые замки (адаптер проверен на 50 реальных замках);
   - боевой магазин YooKassa вместо тестового; боевой email-провайдер.
3. **Инфраструктура на сервере:**
   - managed PostgreSQL + миграции + автобэкапы (уйти с embedded-postgres);
   - docker-контейнер **Collabora Online** + `COLLABORA_URL`/`PUBLIC_API_URL` — онлайн-редактирование офисных файлов Диска (WOPI-хост готов и прогнан curl);
   - **GIN-индексы FTS + русская токенизация** — полнотекстовый поиск КБ «оживает» на проде;
   - Redis/BullMQ вместо БД-очереди синка каналов + Docker (по DHP — опционально, по мере нагрузки).
4. **Ключи фронта от владельца:** `NEXT_PUBLIC_YANDEX_METRIKA_ID` (Метрика), боевой `NEXT_PUBLIC_YANDEX_MAPS_API_KEY`.

### Функциональная очередь (по явному запросу владельца, ядро готово как enabler)
5. Реальные OTA-адаптеры Channel Manager (Ostrovok / Avito / Суточно — сейчас mock).
6. Миграция гостевых web/mobile со старого Bnovo-флоу на собственный Booking Engine (бэкенд готов со Sprint 5).
7. Мобильное приложение: EAS Build, публикация в сторах, Bluetooth eKey TTLock.
8. Боевая фискализация (АТОЛ через FiscalPort) + эквайринг БСПБ/PayKeeper.
9. BI-дашборд / расширенная аналитика (v2), реестр оборудования (v2).
10. Хвост КБ/Диска: rate-limit `/s/:token`, diff версий, квоты Диска, порог релевантности `kb/ask`.
