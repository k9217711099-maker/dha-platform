# ТЗ: Модуль «Задачи и Уборка» (Operations 2.0)

Проектное ТЗ на замену существующих разделов админки **PMS · Уборка** и
**PMS · Обслуживание** единым операционным модулем «Задачи и Уборка» —
по образу и подобию **TeamJet** (справочный центр https://teamjet.com/help
изучен полностью, конспект — §1). Документ рассчитан на реализацию в Claude Code
в этом монорепо, с переиспользованием уже построенного (PMS, склад, RBAC,
уведомления, аудит, SSE-realtime мессенджера).

Статус: **внедрено (MVP, 2026-07-11)** — модуль `apps/api/src/ops/*` +
админ-раздел «Операции» (`apps/admin/src/app/ops/*`); старые разделы PMS·Уборка
и PMS·Обслуживание удалены (маршруты редиректят). Включая: UI списания
расходников (модалка в карточке + листы списания в настройках), привязку
номеров к секциям, группировку плана по этажам/секциям, отчёт «Таймлайн»,
блок номера в карточке задачи (гость по ops_guest_info, DND/«просит уборку»,
история задач номера), лимит месяца на Excel-экспорт списка, а также
v2-блоки: аналитику чек-листов (§5.4), отчёты «Pro» (почасовая загрузка +
шаблоны), печать заданий горничным (§6.3), QR-коды объектов, правила уборок
по тарифу/промокоду (§6.2).

Дополнение 2026-07-13 — **LQA-контур инженерной службы**: SLA-матрица
(критичность × источник гость/внутренняя → автопроставление acceptBy/dueAt,
`OpsSlaPolicy`, настройки → SLA), флаг задачи `guestRequest` («Заявка гостя»),

старта), карточка задачи открывается из панели номера на шахматке.

## 0. Контекст: что есть сейчас и что меняем

### 0.1 Существующий функционал (заменяем/развиваем)

| Сейчас | Где | Что умеет |
|---|---|---|
| PMS · Уборка | `apps/admin/src/app/pms/housekeeping/page.tsx`, `apps/api/src/pms/operations/housekeeping.*` | Плоский список задач уборки: создать (объект+номер+тип), назначить (строка `assignee`), начать, завершить (чисто/проверено). Статусы `TODO → IN_PROGRESS → DONE`. Ведёт `Room.housekeepingStatus` |
| PMS · Обслуживание | `apps/admin/src/app/pms/maintenance/page.tsx`, `apps/api/src/pms/operations/maintenance.*` | Инженерные заявки: заголовок, критичность (MINOR/MAJOR/CRITICAL), `blocksSale` → номер `OUT_OF_ORDER` и выбывает из продажи; start/resolve |
| Авто-задача на выезд | `pms-booking.service.ts` (`checkOut`) | При чек-ауте: номер → `DIRTY` + создаётся `HousekeepingTask(DEPARTURE_CLEAN, bookingId)` |
| Модели | `schema.prisma`: `HousekeepingTask`, `MaintenanceTask`, `Room.housekeepingStatus` (CLEAN/DIRTY/IN_PROGRESS/INSPECTED), `Room.maintenanceStatus` (OK/OUT_OF_ORDER) | |
| Права | `admin/permissions.ts`: `pms_housekeeping`, `pms_maintenance`; роли `pms_hk_supervisor`, `pms_engineer` | |

### 0.2 Что переиспользуем (не строим заново)

- **Склад** (`apps/api/src/warehouse/*`) — инвентаризация и списание расходников
  при уборках (аналог TeamJet «Инвентаризация»/«Списание при уборках») делается
  поверх существующих остатков/движений, свой параллельный учёт НЕ создаём.
- **RBAC** (`admin/permissions.ts`, страница `/roles`) — новые права `ops_*`.
- **Аудит** (`warehouse/audit`) — журналируем весь жизненный цикл задач.
- **Уведомления** (`notifications/*`, `PushPort`) + **SSE-паттерн** мессенджера
  (`staff-chat/staff-chat.stream.controller.ts`) — realtime-обновления списков
  задач и push исполнителям.
- **Сотрудники** — `AdminUser` (+`lastSeenAt` для presence). Исполнители задач —
  это `AdminUser`, а не свободная строка `assignee`, как сейчас.
- **Очереди** — БД + `@nestjs/schedule` (без Redis), как везде в проекте.

### 0.3 Ключевые решения (зафиксировать)

1. **Единая модель `OpsTask`** для всего: обычные задачи, уборки, инженерные
   заявки. Разделение через `kind: TASK | CLEANING` + категории/теги.
   «Обслуживание» перестаёт быть отдельным разделом — это задачи инженерной
   службы (тег «Ремонт»), при этом механика `blocksSale → OUT_OF_ORDER`
   полностью сохраняется как свойство задачи.
2. **Существующие `HousekeepingTask`/`MaintenanceTask` мигрируются** в `OpsTask`
   и удаляются из схемы (данных в проде нет — только dev-сиды; миграция скриптом
   в рамках Prisma migrate).
3. **Мобильный интерфейс исполнителя (MVP) — адаптивная веб-версия админки**
   (горничная открывает админку с телефона; отдельные мобильные экраны «Мои
   задачи»/«Мои уборки» делаем mobile-first). Отдельное приложение сотрудника —
   v2, вопрос владельцу (§14). `apps/mobile` — гостевое, его не трогаем.
4. Тема UI — «Спокойный индиго»/Manrope (действующая тема админки), все
   календари — общий `DateRangePicker` (конвенция проекта).
5. Мультиарендность как везде: все новые таблицы с `tenantId`.

## 3. Терминология и статусные модели

### 3.1 Виды работ

- **Задача** (`kind=TASK`) — универсальное поручение: «поднести утюг в 305»,
  «заменить лампочку», «проверить кондиционер». Может быть без объекта
  (общехозяйственная) или с привязкой к номеру/зоне.
- **Уборка** (`kind=CLEANING`) — специализированная задача с типом уборки,
  нормативом времени, чек-листом типа, инспекцией и влиянием на
  `Room.housekeepingStatus`. Создаётся правилами (§6.2), из плана уборок (§6.3)
  или вручную.

### 3.2 Статусы задачи/уборки (единые, по TeamJet)

```
PLAN ──(наступило scheduledAt)──▶ NEW ─▶ ACCEPTED ─▶ IN_PROGRESS ─▶ DONE
                                   │         │           │  ▲
                                   │         │           ▼  │
                                   │         │         PAUSED
                                   └─────────┴──▶ CANCELLED (причина обязательна)
DONE/CANCELLED ──(переоткрыть, право ops_manage)──▶ NEW
```

| Статус | Значение | Правила |
|---|---|---|
| `PLAN` | создана заранее, исполнителю не видна | не попадает в отчёты; авто-переход в `NEW` по `scheduledAt` (cron раз в минуту) |
| `NEW` | исполнитель не ознакомился | push исполнителю при переходе |
| `ACCEPTED` | принял на себя | фиксируем `acceptedAt` |
| `IN_PROGRESS` | выполняется | фиксируем `startedAt`; для уборки: номер → `IN_PROGRESS` («на уборке») |
| `PAUSED` | отложена, вернётся | время паузы не входит в факт длительности |
| `DONE` | сделана | нельзя, пока чек-лист не завершён; `completedAt`; для уборки: номер → `CLEAN` |
| `CANCELLED` | отменена | обязателен комментарий-причина |

Факт-время работы = сумма интервалов `IN_PROGRESS` (для сравнения с нормативом §6.1).

### 3.3 Статусы номера (расширяем текущие)

`Room.housekeepingStatus` — остаётся enum `CLEAN / DIRTY / IN_PROGRESS /
INSPECTED`; переходы теперь только через задачи уборки и инспекцию.
`Room.maintenanceStatus` — `OK / OUT_OF_ORDER`, двигается задачами с
`blocksSale` (как сейчас).

Новое на номере (по TeamJet `/help/locations`):
- `dndUntil DateTime?` — «Не беспокоить» (с ограничением по времени; ставит
  администратор или, позже, гость из приложения). Пока действует — уборки по
  номеру не отправляются исполнителю (висят с пометкой DND).
- `cleanRequestedAt DateTime?` — «Просит уборку» (поднимает приоритет).

Оба поля видны на шахматке и в плане уборок.

## 4. Раздел «Задачи»

Образец: https://teamjet.com/help/tasks

### 4.1 Поля задачи

| Поле | Тип | Обяз. | Примечание |
|---|---|---|---|
| `title` | строка | да | |
| `description` | текст | нет | URL в тексте кликабельны |
| `propertyId` | ссылка | да | объект размещения |
| `roomId` / `zoneId` | ссылка | нет | номер ИЛИ нежилая зона (§7); может быть пусто |
| `assignees` | AdminUser[] | да* | ≥1 исполнитель; *может назначить себя создатель |
| `watchers` | AdminUser[] | нет | наблюдатели получают уведомления о смене статусов |
| `important` | bool | нет | «Важная» — выделение в списке, сортировка наверх |
| `dueAt` | datetime | нет | срок выполнения; просрочка подсвечивается красным |
| `acceptBy` | datetime | нет | срок принятия в работу |
| `scheduledAt` | datetime | нет | если задано — задача создаётся в статусе `PLAN` |
| `tags` | OpsTag[] | нет | §8.2 |
| `checklists` | Checklist[] | нет | §5; блокируют `DONE` |
| `attachments` | файлы | нет | фото/файлы, хранение как в Диске (S3-совместимое) |
| `requirePhotoResult` | bool | нет | при завершении требуется фото результата; без него — авто-комментарий «требует проверки» |
| `blocksSale` | bool | нет | инженерное: снять номер с продажи (`OUT_OF_ORDER`) до `DONE`; требует `roomId` |
| `severity` | MINOR/MAJOR/CRITICAL | нет | сохраняем от текущего «Обслуживания», влияет на сортировку |

### 4.2 Список задач

Единый экран `/ops/tasks` с табами **«Задачи» / «Уборки»** (как в TeamJet
мобильном) и видом «по группам» (группировка по отделу исполнителя).

Фильтры (все — серверные, через query): статус, исполнитель, наблюдатель,
создатель, объект, номер/зона, теги, важность, наличие чек-листа,
повторяющиеся, дата (диапазон — `DateRangePicker`), просроченные.
Активный фильтр помечается индикатором; кнопка «Сбросить». Быстрый поиск по
названию/описанию/номеру. Экспорт текущей выборки в Excel (лимит — 1 месяц,
более длинные периоды — в Отчётах §9).

Строка списка: важность (стрелка), №/зона, название, теги (цветные), исполнитель,
статус-бейдж, дедлайн (красный при просрочке), иконка чек-листа с прогрессом.

### 4.3 Карточка задачи

Открывается по клику (модалка/панель, как окно брони на шахматке):
- Шапка: статус + доступные переходы (§3.2) кнопками.
- Поля §4.1, история смены статусов (кто/когда), факт-время в работе.
- **Комментарии**: многострочные, URL кликабельны; уведомления участникам о
  новых комментариях. Отмена задачи требует комментария.
- **Вложения**: превью фото, загрузка с телефона (камера).
- Чек-листы с прохождением (§5.3).
- Блок объекта: карточка номера — гость и даты заезда/выезда текущей брони
  (если право `ops_guest_info`), история задач по номеру (с выгрузкой Excel),
  история статусов номера.

### 4.4 Права на изменение

Редактировать/удалять: создатель, руководитель группы исполнителя, роль с
`ops_manage`. Исполнитель меняет только статус, чек-лист, комментарии, фото.
Видимость задач по праву: все / только своей группы / только свои (§10).

### 4.5 Шаблоны задач

`/ops/templates`: преднастроенные задачи (название, описание, объект, теги,
чек-лист, важность, `requirePhotoResult`, дефолтный исполнитель/группа).
Создание задачи из шаблона — один клик + правки. Право `ops_from_template_only`
ограничивает сотрудника созданием только из шаблонов (TeamJet permissions).

### 4.6 Запланированные задачи

Образец: https://teamjet.com/help/scheduled-task. Кнопка «Запланировать» при
создании: указывается `scheduledAt`, задача создаётся в `PLAN`, исполнителю не
видна и в отчёты не входит; cron переводит в `NEW` и шлёт push. До активации
редактируется/отменяется создателем или руководителем.

### 4.7 Планировщик (повторяющиеся)

Образец: https://teamjet.com/help/tasks/planner. Вкладка «Планировщик» в
Задачах: правило = шаблон задачи + расписание:
- ежедневно (время),
- еженедельно (дни недели + время),
- ежемесячно (числа месяца + время).

Cron (`@nestjs/schedule`, раз в минуту) материализует экземпляры `OpsTask` в
статусе `NEW` в назначенный момент (создание идемпотентно: уникальный ключ
`ruleId+firedAt`). В списке задач фильтр «Повторяющиеся». Правило можно
выключить/включить/редактировать; история созданных экземпляров видна из правила.

## 5. Чек-листы

Образец: https://teamjet.com/help/checklists

### 5.1 Конструктор (`/ops/checklists`)

- Название («Инспекция номера», «Ночной аудит»…).
- Иерархия: **заголовки → пункты → подпункты**, drag&drop перестановка.
- Настройки пункта:
  - ответы «Да/Нет» + опциональный третий вариант (текст задаётся, напр. «N/A»);
  - **фото-подтверждение** — обязательное фото при любом ответе
    (.jpg/.jpeg/.png/.webp/.heic/.gif);
  - **исключить из результата** — не влияет на процент выполнения.
- Версионирование лёгкое: прикреплённый к задаче чек-лист — снапшот пунктов на
  момент прикрепления (правки шаблона не ломают историю).

### 5.2 Привязка

К задаче (в карточке/шаблоне/правиле планировщика) и к **типу уборки** (§6.1,
автоматически вешается на каждую уборку этого типа). Опция типа уборки
«чек-лист перед началом» — нельзя перейти в `IN_PROGRESS`, пока не пройден.

### 5.3 Прохождение исполнителем

- Прогресс в процентах (подпункты в расчёт не входят — как в TeamJet).
- Иконка камеры на пункте: серая (не отмечен) / красная (ответ без фото) /
  зелёная (фото загружено).
- К пункту: комментарий, фото, **создать задачу из пункта** (например,
  «нет — кран течёт» → задача инженеру; поддерживается шаблон по умолчанию).
- Задачу нельзя перевести в `DONE`, пока чек-лист не завершён.
  **Автозавершение**: по кнопке «Завершить» система предлагает автоматически
  дозакрыть неотмеченные пункты — только если все пункты с фото и все с доп.
  вариантом уже отмечены.

### 5.4 Отчёты по чек-листам

- Ответы «Нет» = **ошибки** — попадают в отчёт по уборкам (§9.3).
- Опция «отправлять отчёт о завершении» выбранным сотрудникам (email, .csv) — v2.
- Аналитика (средний % выполнения, динамика) — v2.

## 6. Уборка

Образцы: https://teamjet.com/help/cleaningplan, https://teamjet.com/help/cleaningplan/settings

### 6.1 Справочник «Типы уборок» (`/ops/cleaning/settings`)

Пресеты для жилых: **Выездная, Текущая, Жилая** (переименовываемые) + свои
(«Генеральная», «Смена белья», «Регулярная+шампанское»…). Отдельный набор для
нежилых зон. Поля типа: название, для жилых/нежилых, чек-лист (+флаг «перед
началом»), цвет для плана/шахматки.

**Стандарты (нормативы времени)**: матрица «тип уборки × категория номера» →
минуты. Используются в плане (суммарная загрузка горничной) и в отчётах
(план/факт, красная подсветка превышения).

### 6.2 Правила автосоздания уборок

Заменяют текущее «одно правило» (чек-аут → DEPARTURE_CLEAN) на настраиваемый
набор. Правило: **условие → тип уборки**.

Базовые состояния номера на дату (вычисляются из броней PMS):
1. «Сегодня выезд» — выезд без нового заезда;
2. «Сегодня заезд» — заезд в свободный номер;
3. «Сегодня выезд под заезд» — выезд и заезд в один день;
4. «Свободен» — нет проживания;
5. «Заселён» — гость живёт (не день заезда/выезда).

Доп. условия правила: категория номера (разные сценарии для «Эконом» и
«Люкс»), длительность проживания ≥ N ночей (для «Заселён» — жилая уборка
каждые N дней), тариф/промокод (v2). При совпадении нескольких правил
побеждает более специфичное (больше условий). Пресет после миграции: «Сегодня
выезд → Выездная» (сохраняет текущее поведение чек-аута).

Генерация: ночной cron строит уборки на день по правилам + мгновенная реакция
на события (чек-аут → сразу уборка, как сейчас). Дубликаты не создаются
(уникальность `roomId+date+type` для авто-созданных).

### 6.3 План уборок (`/ops/cleaning/plan`)

Экран распределения на дату (по умолчанию сегодня):
- **Слева** — несозданные/нераспределённые уборки: список номеров с фильтрами
  (статус номера, корпус, этаж, секция, категория), бейджи «выезд/заезд/живёт»,
  DND, «просит уборку».
- **Справа** — колонки горничных «в смене» (+супервайзеры смены). На колонке:
  суммарное нормативное время, число номеров.
- **Распределение**: drag&drop номеров на горничных (по корпусам или по
  секциям); перенос между горничными; изменение типа уборки и приоритета
  (порядок в колонке = порядок выполнения); удаление из плана.
- **Автораспределение**: параметры — участвующие горничные, супервайзеры,
  норматив загрузки (минут/уборок на человека), группировка (по корпусам/этажам
  или секциям), приоритет (сначала выезды под заезд и т.п.); распределяет
  поровну с учётом нормативов.
- **«Отправить задания»** — до отправки уборки в статусе `PLAN` (горничные их
  не видят); отправка переводит в `NEW` + push. Отмена — снять все
  незавершённые уборки (у всех или у конкретного сотрудника).
- Альтернатива: из списка номеров (§7) отметить галочками → «Новая уборка»
  (исполнитель, супервайзер, тип, чек-лист).

### 6.4 Выполнение и инспекция

- Горничная ведёт уборку по статусам §3.2; `IN_PROGRESS` → номер «на уборке»,
  `DONE` → номер `CLEAN`.
- **Инспекция**: если у уборки задан супервайзер (или тип требует проверки) —
  после `DONE` супервайзеру создаётся задача «Инспекция №X» (kind=TASK, тег
  «Инспекция», чек-лист инспекции опционально). Приём: номер → `INSPECTED`;
  отклонение: комментарий + новая уборка-доработка тому же исполнителю, номер
  остаётся `DIRTY`. Упрощённый путь (без задачи-инспекции) — кнопка
  «Проверено» на самой уборке, как сейчас (право `ops_inspect`).
- DND: пока `dndUntil` активен, уборка помечается и не пушится; по истечении —
  напоминание исполнителю.

### 6.5 Связь с PMS

- Чек-аут брони: как сейчас — номер `DIRTY` + мгновенная уборка по правилу
  «Сегодня выезд» (та же транзакция, `bookingId` проставляется).
- Revert check-in / reopen checkout: авто-созданная незавершённая уборка
  отменяется (комментарий «бронь переоткрыта»).
- Шахматка: статус уборки номера уже отображается; добавить бейджи DND и
  «просит уборку», клик по статусу — быстрый переход к активной уборке номера.

### 6.6 Списание расходников при уборке

Образец: https://teamjet.com/help/inventory-cleanings. Реализация — поверх
склада (`warehouse`), без параллельного учёта:
- **Листы списания**: справочник «что списывается при типе уборки» (позиции
  номенклатуры склада + количество по умолчанию), привязка к типу уборки и/или
  категории номера.
- В карточке уборки/номера — «Списать расходники»: предзаполненный лист,
  корректировка количества, подтверждение → создаётся складское движение
  списания (причина `USED`) с привязкой `opsTaskId` со склада-адреса объекта.
- Списание по запросу (вне уборки) — из карточки номера.
- Отчёт: история списаний по номеру/сотруднику/периоду — в складских отчётах,
  фильтр по `opsTaskId`/объекту.

## 7. Объекты уборки: номера и нежилые зоны

Образец: https://teamjet.com/help/locations. Номера (`Room`) уже есть —
корпус/этаж через `floor`/`address`. Добавляем:

- **`Zone`** (нежилые объекты): лобби, ресторан, коридор 3 этажа, офис —

- Карточка объекта (номер/зона): статусы, DND/просьба уборки, активные и
  исторические задачи (Excel-выгрузка), история статусов, списания расходников,
  гость и даты (по праву `ops_guest_info`).

## 8. Автоматизация и теги

Образец: https://teamjet.com/help/settings/automation

### 8.1 Правила автоматизации (`/ops/settings` → «Автоматизация»)

Два типа правил (вкл/выкл переключателем, редактирование, удаление):

1. **Напоминание**: «если задача группы G в статусе S дольше N минут —
   уведомлять исполнителя каждые M минут». Пример TeamJet: HSKP, `NEW` > 30
   мин → пинг раз в 5 мин.
2. **Эскалация**: «если задача группы G в статусе S дольше N минут —
   переназначить на X (сотрудник или руководитель группы) и уведомить».

Движок: cron раз в минуту, состояние «последнего срабатывания» на задаче
(таблица `OpsAutomationFire`), чтобы не спамить. Уведомления — push + бейдж
в админке (SSE).

### 8.2 Теги (`/ops/settings` → «Теги»)

`OpsTag`: название («VIP», «Поломка», «Инспекция»), цвет, комментарий;
архивирование/восстановление. Используются в фильтрах и отчётах. Системные
теги при миграции: «Ремонт» (для бывших MaintenanceTask), «Инспекция».

## 9. Отчёты (`/ops/reports`)

Образцы: https://teamjet.com/help/reports, https://teamjet.com/help/reports/cleanings

1. **Дашборд**: на сегодня — задачи по статусам, номера по housekeeping-статусам,
   загрузка номерного фонда, просроченные, «в ремонте»; фильтр жилые/нежилые.
2. **Задачи за период**: создано/выполнено/отменено по дням (график + таблица),
   фильтры (теги, группы, сотрудники, объекты), клик по цифре → список,
   экспорт Excel. Среднее время реакции (NEW→ACCEPTED) и выполнения
   (IN_PROGRESS суммарно).
3. **Уборки**: за период по сотрудникам — количество и типы уборок, номера,
   факт-время vs норматив (**превышение — красным**), среднее время, инспектор,
   **ошибки чек-листа** (пункты «Нет»). Группировка: исполнитель / тип уборки /
   категория номера. Персональный лист горничной на день. Экспорт Excel.
4. **Таймлайн** (v2): время задач по статусам с дневным сравнением в %.

Статус `PLAN` в отчёты не входит (как в TeamJet).

## 10. RBAC

Новые права в `admin/permissions.ts` (`ops_*`); `pms_housekeeping` и
`pms_maintenance` удаляются, пресеты ролей обновляются миграцией:

| Право | Что даёт |
|---|---|
| `ops_tasks` | раздел Задачи/Уборки: видеть **свои** задачи, вести статусы, чек-листы, комментарии, списания |
| `ops_view_group` / `ops_view_all` | видимость задач своей группы / всех (TeamJet: три уровня видимости) |
| `ops_create` | создавать задачи (без него + `ops_from_template_only` — только из шаблонов) |
| `ops_manage` | редактировать/переназначать/удалять/переоткрывать чужие задачи |
| `ops_cleaning_plan` | экран «План уборок», распределение и отправка заданий |
| `ops_inspect` | инспекция уборок («Проверено», номер → INSPECTED) |
| `ops_checklists` | конструктор чек-листов |
| `ops_settings` | типы/правила уборок, стандарты, автоматизация, теги, листы списания |
| `ops_reports` | отчёты |
| `ops_guest_info` | видеть гостя и даты в карточке объекта/задачи |

Обновление пресетов ролей: `pms_hk_supervisor` → + `ops_tasks, ops_view_group,
ops_create, ops_manage, ops_cleaning_plan, ops_inspect, ops_reports`;
`pms_engineer` → + `ops_tasks, ops_view_group, ops_create`; `pms_owner`/`pms_gm`
→ все `ops_*`. Новая роль-пресет **«Горничная»** (`ops_maid`): `ops_tasks`.

Режим **«В смене»**: флаг на `AdminUser` (`onDutySince DateTime?`),
переключается сотрудником в своём профиле и руководителем в списке персонала.
Не «в смене» — не получает push и не участвует в автораспределении. План
уборок показывает только сотрудников в смене.

Группы (отделы) — используем существующую модель ролей/`Role`; если отдельной
сущности «группа» нет, в MVP группой считается роль сотрудника (HSKP =
роль «Горничная»+«Супервайзер HSKP»). Полноценные отделы — вместе с
оргструктурой (вопрос §14).

миграции данных (маппинг: HK `TODO→NEW`, `IN_PROGRESS→IN_PROGRESS`,
`DONE→DONE`, type→CleaningType-пресет; MT `OPEN→NEW`, `RESOLVED→DONE`,
тег «Ремонт», `blocksSale` переносится).

## 13. Этапы реализации

1. **Ядро задач**: модели `OpsTask`+спутники, миграция HK/MT, статусная машина,
   CRUD+фильтры, комментарии, вложения, права `ops_*`, редиректы, аудит.
   Паритет со старыми разделами достигается здесь (включая `blocksSale`).
2. **Чек-листы**: конструктор, снапшоты, прохождение, блокировка `DONE`,
   автозавершение, задача из пункта.
3. **Уборка**: типы/стандарты/правила, генерация, план уборок (ручное
   распределение + отправка), инспекция, DND/просьба уборки, шахматка.
4. **Автораспределение + автоматизация + планировщик**: авторазброс по
   нормативам, напоминания/эскалации, повторяющиеся и запланированные задачи,
   шаблоны.
5. **Расходники + отчёты + мобильный экран**: списание при уборке, отчёты
   §9 (дашборд, задачи, уборки план/факт), `/ops/my`, режим «в смене»,
   SSE + push, Excel-экспорты.

Каждый этап — рабочее состояние (деплоится независимо); e2e-сценарий в
`e2e-dhp.mjs` дополняется на этапах 1 и 3.

## 14. Критерии приёмки (MVP)

1. Чек-аут брони → номер `DIRTY`, уборка «Выездная» создана; горничная в смене
   получает её после «Отправить задания», ведёт `NEW→…→DONE` с чек-листом и
   фото; номер `CLEAN`; супервайзер жмёт «Проверено» → `INSPECTED`. Факт-время
   и ошибки чек-листа видны в отчёте «Уборки», превышение норматива — красным.
2. Инженерная заявка с `blocksSale` снимает номер с продажи (нет в
   availability), `DONE` возвращает; вся история — в карточке и аудите.
3. Задача с чек-листом не закрывается до завершения пунктов; пункт с фото
   требует фото; отмена — только с комментарием; переоткрытие работает.
4. Запланированная задача активируется в срок; повторяющаяся создаётся по
   расписанию без дублей; эскалация переназначает и уведомляет.
5. Правила уборок: «выезд под заезд» и «жилая каждые 3 ночи» создают верные
   типы; при пересечении побеждает более специфичное правило.
6. Права: горничная видит только свои задачи; `ops_view_group` — группу;
   без `ops_guest_info` данные гостя скрыты. Старые права/маршруты удалены,
   роли-пресеты обновлены.
7. Списание при уборке уменьшает складские остатки и видно в складской
   истории с привязкой к задаче.
